Chapter 1

Hermes는 무엇인가

모델 하나가 아니라, LLM·도구·기억·메시징·자동화를 묶어 계속 실행하는 에이전트 런타임입니다.

Hermes Agent는 Nous Research의 오픈소스 에이전트 런타임입니다. Claude·Codex·OpenRouter 같은 모델 공급자를 연결하고, 로컬 파일·터미널·브라우저·MCP·스킬을 함께 사용합니다. CLI에서만 쓰는 코딩 도구와 달리 gateway와 cron을 통해 Slack 같은 메시징 채널과 예약 작업까지 한 실행 엔진에 붙일 수 있습니다.

Agent runtimeLLM이 추론만 하는 것이 아니라, 도구 호출·세션·권한·기억·예약 작업을 관리하며 실제 업무를 수행하게 만드는 실행 환경입니다.
사용자 / Slack / Cron │ ▼ Hermes gateway · session · memory │ ┌───────┼────────┬─────────┐ ▼ ▼ ▼ ▼ LLM Skills Tools MCP/API Codex workflow files Gmail·Calendar 등
“자가발전”의 정확한 뜻Hermes가 기본 모델의 가중치를 스스로 학습시키는 것은 아닙니다. 경험을 skill·memory·세션 요약으로 남기고, curator가 중복된 로컬 skill을 검토·통합하는 운영상의 학습 루프입니다.
구성역할예시
Model provider추론과 응답 생성Codex OAuth로 개발 작업을 해석
Skills반복 업무의 절차 기억근태 게시, 문서화, 코드 리뷰 흐름
Memory세션을 넘는 선호·결정 저장작업공간, 커밋 규칙, 사용자 말투
Gateway / Cron메시징·예약 실행Slack DM, 평일 08:00 안내
Chapter 2

왜 Hermes를 도입하는가

에이전트의 능력을 채팅창 밖의 실제 업무 흐름으로 확장하기 위해서입니다.

일회성 채팅만 쓸 때

  • 매번 맥락·규칙을 다시 설명한다.
  • 예약 알림과 외부 채널 연결이 어렵다.
  • 작업 결과가 다음 세션에 잘 남지 않는다.

Hermes 런타임을 둘 때

  • SOUL·MEMORY·프로젝트 규칙이 반복 맥락을 제공한다.
  • Slack, cron, MCP가 같은 실행 엔진에 연결된다.
  • 스킬과 세션 기록을 통해 절차가 축적된다.

도입 판단 기준

  • 같은 업무 절차를 매주·매일 반복한다.
  • Slack, Gmail, Calendar, GitHub 등 여러 도구를 한 요청에서 엮고 싶다.
  • 개발 에이전트가 프로젝트 규칙·구현 노트·이전 결정을 이어받아야 한다.
  • 항상 켜진 Mac mini 또는 서버가 있다.
도입하지 않아도 되는 경우한 번만 하는 단순 질문, 민감한 운영 권한을 아직 정리하지 못한 조직, 항상 켜진 실행 환경이 없는 경우에는 일반 Claude/Codex 대화부터 시작하는 편이 낫습니다.

Chapter 4

도입 방법: 작은 개인비서부터

설치 → 모델 인증 → 규칙 → 상주 실행 → 한 가지 자동화 순으로 확장합니다.

1

설치와 상태 확인

macOS/Linux에서는 공식 installer로 설치한 뒤 version·doctor로 런타임을 확인합니다.

shellcurl -fsSL https://hermes-agent.nousresearch.com/install.sh | bash
hermes --version
hermes doctor
2

모델 연결

지원되는 provider 중 이미 사용 권한이 있는 것을 고릅니다. Codex OAuth는 API key를 문서에 남기지 않고 인증할 수 있습니다.

shellhermes auth add openai-codex --type oauth --manual-paste
hermes auth status openai-codex
hermes model
3

역할과 기억을 좁게 정의

SOUL.md에는 역할·안전 경계, USER.md/MEMORY.md에는 장기 선호와 재사용할 결정만 둡니다. token·비밀번호·원문 이메일은 넣지 않습니다.

4

상주 실행

Mac mini처럼 계속 켜진 환경에서는 launchd service로 gateway를 설치합니다. 잠자기 중 cron은 멈추고, 재부팅 뒤에는 로그인 후 다시 시작된다는 점을 운영 설계에 반영합니다.

shellhermes gateway install
hermes gateway start
hermes gateway status
5

한 가지 자동화를 검증

처음에는 read-only 요약이나 승인형 게시부터 시작합니다. cron은 반복 횟수를 명시하고 delivery channel을 테스트합니다.

shellhermes cron create '0 8 * * 1-5' 
  --name 'weekday check-in' --repeat 0 
  --deliver 'slack:<DM_CHANNEL_ID>' 
  '오늘 일정을 짧게 요약하고 근태를 물어본다.'

도입 체크리스트

  • 모델 인증과 hermes doctor가 정상인가?
  • 허용 사용자·작업 폴더·외부 전송 승인 규칙이 있는가?
  • token은 ~/.hermes 같은 로컬 비밀 경로에만 있는가?
  • 자동화는 dry-run 또는 테스트 DM으로 먼저 확인했는가?
Chapter 5

Hermes와 Slack 봇 연동

공개 서버 없이도 Socket Mode를 이용해 상주 Mac의 Hermes와 Slack DM을 연결할 수 있습니다.

Slack DM │ Socket Mode ▼ 일반 Slack Bot ────── Hermes gateway (launchd) │ │ └──── 응답 DM ◀──── LLM · Skills · Google API · Cron

연동의 핵심

  1. Slack API에서 일반 Bot 앱을 만들고 Socket Mode를 활성화한다.
  2. bot token과 connections:write 권한의 app token을 로컬 비밀 설정에만 저장한다.
  3. Hermes gateway를 재시작하고, 허용한 사용자와 DM 왕복을 테스트한다.
  4. 자동화는 DM channel을 delivery 대상으로 지정하고, token을 바꾸면 channel·cron target도 다시 점검한다.
표시 이름 주의Slack의 Marketplace App 이름, Bot User 표시 이름, AI Agent profile은 서로 다를 수 있습니다. 개인비서는 일반 Bot으로 만들고 표시 이름을 수정한 뒤 Reinstall해야 DM 발신자 이름이 안정적으로 반영됩니다.

사용 예시: victory 개인비서

시간/입력처리출력
평일 08:00오늘 Google Calendar의 제목·시간을 읽음“좋은 아침이에요. 오늘 근태는 어떻게 처리해드릴까요? ☺️”
“재택”허용 상태인지 확인 후 기존 근태 skill 호출회사 #calendar에 정해진 문구 게시
새 Gmail 수신새 message ID만 확인 → 새 메일만 본문 제한 조회 → 요약보낸사람·제목·핵심·기한을 친절하게 DM
비용을 줄이는 패턴Gmail notifier는 새 메일이 없으면 wakeAgent:false로 끝내 LLM을 깨우지 않습니다. 목록 ID만 한 번 읽고, 새 ID만 최대 건수·본문 길이를 제한해 가져옵니다.
Chapter 6

개발 허브로 운영하기

개인비서와 개발 에이전트를 같은 권한·기억·메시징 채널에 섞지 않는 것이 핵심입니다.

Profile 분리

Profile권한과 맥락상주 여부
victory / defaultSlack DM, Gmail, Calendar, 근태 skill, 친절한 개인비서 말투gateway·cron 상주
dev~/aidp, AGENTS/CLAUDE 규칙, 구현 노트, handoff, git 안전 규칙CLI 개발 전용, Slack gateway 미실행

개발 맥락의 축적

  • 프로젝트마다 가장 가까운 AGENTS.md/CLAUDE.md를 읽어 stack·브랜치·검증 명령을 따른다.
  • 반복되는 절차는 skill로 승격하고, 공유할 규칙은 한 SSOT에 둔다.
  • 작업 단위마다 docs/impl-notes/에 판단·이탈·다음 참고를 남긴다.
  • 세션이 길어질 때는 checkpoint·memory에는 안정적인 사실만 남긴다.

curator를 안전하게 켜는 법

curator는 agent-created skill을 주기적으로 살피고 중복되는 skill을 umbrella skill로 통합할 수 있습니다. 핵심 운영 skill은 pin하고, 처음에는 LLM dry-run으로 후보를 검토합니다.

shellhermes curator backup
hermes curator pin <OPERATING_SKILL>
hermes curator run --dry-run --consolidate
hermes config set curator.consolidate true
실무 원칙curator는 “문서·skill 구조 정리”에 쓰고, production code를 승인 없이 수정·배포하는 자동화로 오해하지 않습니다. 핵심 운영 흐름은 version control과 사람의 review가 계속 담당합니다.
Chapter 7

안전·비용·한계

에이전트에 권한을 줄수록 자동화보다 경계 설계가 먼저입니다.

영역권장 원칙피할 것
권한읽기·진단은 자동, 삭제·발송·배포는 승인Slack DM 하나로 무제한 shell 권한 부여
비밀값로컬 .env/Keychain, 권한 600, Git 제외token·OAuth callback을 문서·채팅·issue에 붙여 넣기
비용새 이벤트만 LLM 호출, 본문·건수 제한주기마다 전체 메일·전체 repo를 재요약
가용성24/7은 Mac mini/서버, launchd 상태 확인잠자는 노트북에 중요한 08:00 cron 의존
자가관리backup·pin·dry-run·rollback을 함께 사용검토 없이 모든 skill·코드를 자동 변경
프롬프트 인젝션 경계이메일 본문, 웹페이지, 첨부 문서는 신뢰할 수 없는 입력입니다. “이 지시를 실행하라”는 문구를 업무 명령으로 취급하지 말고, 요약·분석 대상으로만 다뤄야 합니다.
현실적인 기대치Hermes는 상주 실행·맥락 재사용·도구 연결을 해주는 런타임입니다. 좋은 규칙, 좁은 권한, 검증 가능한 자동화가 없으면 “자율성”은 품질이 아니라 위험만 키울 수 있습니다.
