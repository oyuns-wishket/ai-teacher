Chapter 1

하네스 엔지니어링 기초 — 왜 Hook인가?

AI Agent를 감싸는 제어 구조의 본질과, Hook이 왜 필수적인 구성 요소인지를 이해합니다.

1.1 하네스(Harness)란 무엇인가

Harness — 직역: 마구(말을 제어하는 장비). AI 맥락: 모델을 감싸서 제어하는 전체 인프라
AI 모델이 에이전트로서 기능할 수 있도록 감싸는 전체 시스템. 시스템 프롬프트, 도구 정의, 인프라, 오케스트레이션 로직, 제약 조건 등을 모두 포함합니다.
"Agent = Model + Harness. If you're not the model, you're the harness." — Viv Trivedy, AI 엔지니어. Martin Fowler 팀의 Birgitta Bockeler가 재인용

이 공식이 의미하는 바는 명확합니다. AI 에이전트는 LLM 모델 하나만으로 완성되지 않습니다. 모델을 어떤 맥락에서, 어떤 도구와 함께, 어떤 제약 아래에서 실행하느냐가 에이전트의 품질을 결정합니다. 이 "나머지 전부"를 하네스라고 부릅니다.

1.2 왜 2026년은 "하네스의 해"인가

2025년까지는 더 좋은 모델을 만드는 것이 경쟁의 핵심이었습니다. 하지만 GPT-4o, Claude Sonnet, Gemini Pro 등 주요 모델의 성능이 수렴하면서, 차별화의 축이 이동했습니다.

"The model is commodity. The harness is moat(해자)." — Aakash Gupta, 프로덕트 전략가
Moat — 직역: 해자(성을 둘러싼 물길). 비즈니스 맥락: 경쟁자가 쉽게 따라올 수 없는 구조적 경쟁 우위
모델은 누구나 API로 접근 가능하지만, 수천 시간의 엔지니어링이 투입된 하네스는 쉽게 복제할 수 없다는 의미입니다.

실제로 Terminal Bench 벤치마크에서, 동일한 모델이 하네스 설정(Hook 구성 포함)만 바꿔서 30위에서 5위로 상승한 사례가 보고되었습니다(Addy Osmani, Google Chrome 팀). 모델을 바꾸지 않고도, 하네스만으로 극적인 성능 차이를 만들 수 있다는 실증입니다.

1.3 Hook의 위치와 역할

하네스 구성 요소 중에서 Hook(직역: 갈고리. 실행 흐름의 특정 지점에 "걸어두는" 사용자 정의 로직)은 에이전트 실행 라이프사이클의 특정 지점에서 자동으로 실행되는 사용자 정의 로직입니다.

소프트웨어 개발에서 Hook 개념은 새롭지 않습니다:

영역Hook 예시하는 일
Git pre-commit, pre-push 커밋/푸시 전에 린트, 테스트 자동 실행
React useEffect, useState 컴포넌트 라이프사이클에 사이드 이펙트 연결
Webpack Tapable Hooks 빌드 파이프라인의 각 단계에 플러그인 로직 삽입
Next.js middleware.ts 라우트 처리 전에 인증, 리다이렉트, 로깅 등 수행
AI Agent PreToolUse, onStepFinish 에이전트가 도구를 호출하기 전/후에 검증, 차단, 로깅

AI 에이전트의 Hook은 이 전통적 개념을 확장하여, 비결정론적(확률적)인 LLM의 행동을 결정론적인 시스템 제약으로 통제하는 핵심 메커니즘으로 자리잡았습니다.

Chapter 2

Hook의 핵심 원리 — 확률적 준수 vs 결정론적 강제

Hook이 단순한 콜백을 넘어 왜 "엔지니어링 패러다임"이 되는지, 그 사상적 기반을 깊이 이해합니다.

2.1 두 가지 준수 모델

Probabilistic Compliance — 확률적 준수. 모델이 지시를 "아마도" 따를 것이라는 기대에 의존하는 방식
프롬프트로 "이렇게 해라"고 지시하는 방식. 모델이 대부분 따르지만, 100% 보장되지 않습니다.
Deterministic Enforcement — 결정론적 강제. 시스템 수준에서 위반 자체를 물리적으로 불가능하게 만드는 방식
Hook이나 하드코딩된 검증으로 제약을 강제하는 방식. 모델이 무엇을 시도하든, 위반은 실행 전에 차단됩니다.
확률적 준수 (프롬프트만 사용)
시스템 프롬프트:
"절대로 rm -rf 명령을 실행하지 마세요.
프로덕션 데이터베이스에 DROP TABLE을
실행하지 마세요."

→ 모델이 99% 따르지만,
  1%의 실패가 치명적일 수 있음
→ 컨텍스트가 길어지면 "잊어버림"
→ Prompt Injection으로 우회 가능
결정론적 강제 (Hook 사용)
PreToolUse Hook:
if (command.includes("rm -rf")) {
  return { decision: "deny" };
}
if (command.includes("DROP TABLE")) {
  return { decision: "deny" };
}

→ 모델이 무엇을 시도하든 100% 차단
→ 컨텍스트 길이와 무관
→ Prompt Injection을 무력화
핵심 통찰 "I told the agent to do X(에이전트에게 X를 하라고 말했다)" vs "The system enforces X(시스템이 X를 강제한다)" — 이 차이가 프로덕션 환경에서의 신뢰성을 결정합니다. Hook은 기대(expectation)를 보장(guarantee)으로 전환합니다.

2.2 왜 프롬프트만으로는 부족한가

2025년 Robust Intelligence의 연구에 따르면:

  • 프로덕션 LLM 애플리케이션의 78%가 최소 1가지 프롬프트 인젝션에 취약
  • 멀티 스텝 워크플로에서 단계당 85% 정확도라도, 10단계를 거치면 전체 성공률은 0.85^10 ≈ 20%
  • 컨텍스트 윈도우에만 의존한 상태 유지는 단계당 약 2%씩 상태가 손실됨
실제 사고 사례 — Replit DROP DATABASE 에이전트에게 "데이터베이스를 수정하지 마라"는 지시가 있었지만, 에이전트는 DROP DATABASE를 실행했습니다. 승인 게이트(Approval Gate) 없이 완전한 자율성을 부여한 것이 원인이었습니다. 이것이 All-or-Nothing Autonomy(전부 아니면 전무 자율성) Anti-Pattern의 대표 사례입니다.

2.3 Martin Fowler 팀의 제어 분류 — Guides와 Sensors

Martin Fowler 팀의 Birgitta Bockeler는 하네스의 제어를 두 가지로 분류합니다:

Guides (Feedforward) — 가이드(사전 제어). 에이전트가 행동하기 전에 방향을 잡아주는 제어
에이전트의 행동을 예측하고, 행동 전에 조향(steer)하는 메커니즘. 예: 시스템 프롬프트, 규칙 파일, PreToolUse Hook
Sensors (Feedback) — 센서(사후 피드백). 에이전트가 행동한 후에 관찰하고 교정하는 제어
에이전트의 행동 후 결과를 관찰하고 자기 교정을 유도하는 메커니즘. 예: PostToolUse Hook, 테스트 실행, 린트 체크

2.4 인간의 역할 재정의

"Humans steer. Agents execute."
(인간이 방향을 잡고, 에이전트가 실행한다.) — Ryan Lopopolo, OpenAI

하네스 엔지니어링 관점에서 인간의 역할은 "직접 코드를 작성하는 사람"에서 "하네스를 설계하고 조율하는 사람"으로 전환됩니다. 문제가 발생할 때마다 모델을 탓하는 대신, 하네스(특히 Hook)를 개선하는 것이 올바른 접근입니다.

The Ratchet Pattern (래칫 패턴) — 래칫: 한 방향으로만 회전하는 톱니바퀴. "한번 개선하면 절대 퇴보하지 않는" 패턴을 비유
Addy Osmani(Google)가 제시한 핵심 개발 방법론입니다:

1) 에이전트가 실수를 함 (예: 주석 처리된 테스트를 커밋)
2) 규칙 파일에 지침 추가 → 확률적 준수
3) Hook으로 패턴을 자동 감지/차단 → 결정론적 강제
4) 같은 실수는 영구적으로 불가능해짐

이렇게 각 실패가 영구적 규칙으로 전환되어, 시스템이 단조 증가(monotonically improving)합니다.

3.3 Augment Code의 3계층 아키텍처

실전에서 Hook을 구성할 때 가장 유용한 프레임워크입니다:

Layer 1: Constraint Harness (제약 하네스) — Feedforward — 사전에 솔루션 공간을 줄여서 에이전트가 실수할 여지를 제거

규칙 파일, 린터, 타입 시스템이 생성 전에 제약을 설정합니다. OpenAI는 이를 taste invariants(취향 불변량 — 팀의 코딩 표준을 경고가 아닌 CI 실패로 강제하는 규칙)라고 부릅니다.

구현: CLAUDE.md, AGENTS.md, ESLint 규칙, TypeScript strict mode, PreToolUse Hook
Layer 2: Feedback Loop (피드백 루프) — Corrective — 에러 메시지가 자동으로 에이전트의 컨텍스트에 재주입되어 자기 교정을 유도

핵심 원리: 에러 메시지 자체가 프롬프트가 됩니다. 린트 에러가 "use logger.info({event: 'name', ...data})"처럼 구체적이면, 에이전트는 이를 읽고 스스로 수정합니다.

구현: PostToolUse Hook (lint/test 자동 실행), onStepFinish 콜백, CI 에러 피드백
Layer 3: Quality Gate (품질 게이트) — Enforcement — 결정론적 CI 게이트가 비준수 결과물의 머지/배포를 물리적으로 차단

표준 CI 파이프라인은 에이전트 특유의 실패(아키텍처 드리프트, 의존성 전략 불일치 등)를 감지하지 못할 수 있으므로, 에이전트 전용 검증 게이트가 필요합니다.

구현: Stop Hook (완료 조건 미충족 시 종료 차단), CI/CD 게이트, 코드 리뷰 서브에이전트

5가지 Hook 핸들러 타입

Claude Code는 다양한 실행 방식을 지원합니다:

1. Command Hook — 셸 스크립트를 직접 실행. 가장 기본적이고 범용적

// .claude/settings.json
{
  "hooks": {
    "PreToolUse": [{
      "matcher": "Bash",              // Bash 도구 호출 시에만 작동
      "hooks": [{
        "type": "command",             // 셸 명령 실행 타입
        "command": "./hooks/validate-bash.sh"
      }]
    }]
  }
}

2. HTTP Hook — 외부 웹서버에 POST 요청. 중앙 집중식 검증 서버 운영 시 유용

{
  "type": "http",
  "url": "http://localhost:8080/hooks/pre-tool-use",
  "headers": {
    "Authorization": "Bearer $MY_TOKEN"   // 환경 변수 참조 가능
  }
}

3. MCP Tool Hook — 연결된 MCP 서버의 도구를 호출. 외부 시스템과의 통합에 적합

{
  "type": "mcp_tool",
  "server": "my_security_server",   // MCP 서버 이름
  "tool": "security_scan",           // 호출할 도구
  "input": { "file_path": "${tool_input.file_path}" }
}

4. Prompt Hook — LLM에게 yes/no 판단을 요청. 규칙으로 표현하기 어려운 뉘앙스 판단에 적합

{
  "type": "prompt",
  "prompt": "Is this bash command safe to execute? $ARGUMENTS",
  "model": "claude-haiku-4-5"        // 빠른 모델 사용 권장
}

5. Agent Hook — 도구 접근 권한이 있는 서브에이전트를 생성. 복잡한 검증 시나리오에 적합

{
  "type": "agent",
  "prompt": "Verify this file edit follows our architecture patterns: $ARGUMENTS"
  // 서브에이전트가 파일을 읽고, grep을 실행하고, 판단을 내림
}

Exit Code와 JSON 출력 프로토콜

Exit Code — 종료 코드. 프로세스가 종료될 때 반환하는 정수값으로, 성공/실패를 시스템에 알림
Claude Code Hook에서: 0 = 성공, 2 = 차단(에러를 에이전트에 전달), 그 외 = 비차단 에러(로깅만)
#!/bin/bash
# hooks/block-dangerous-commands.sh
# stdin으로 도구 호출 정보가 JSON으로 전달됨

COMMAND=$(jq -r '.tool_input.command' < /dev/stdin)

if echo "$COMMAND" | grep -qE 'rm -rf|DROP TABLE|git push --force'; then
  # JSON으로 차단 결정을 반환
  jq -n '{
    hookSpecificOutput: {
      hookEventName: "PreToolUse",
      permissionDecision: "deny",
      permissionDecisionReason: "위험한 명령이 감지되었습니다"
    }
  }'
  exit 0   # exit 0 + deny JSON = 도구 실행 차단
else
  exit 0   # exit 0 + no JSON = 정상 진행
fi
서브에이전트 재귀 적용 (Recursive Hook Enforcement) — 서브에이전트(하위 에이전트)가 생성되어도, 부모의 PreToolUse/PostToolUse Hook이 자동으로 적용됨

이것이 중요한 이유: 서브에이전트를 통한 우회(bypass)를 방지합니다. 만약 Hook이 서브에이전트에 적용되지 않는다면, 에이전트는 서브에이전트를 생성하여 위험한 명령을 실행할 수 있습니다.

4.2 OpenAI Agents SDK Lifecycle Hooks

OpenAI Agents SDK는 Python 클래스 기반의 Hook 시스템을 제공합니다. RunHooksBase와 AgentHooksBase 두 계층으로 구성됩니다.

RunHooksBase — 실행(Run) 수준의 Hook 기반 클래스. 전체 에이전트 실행을 관통하는 콜백
에이전트 실행의 전체 수명 동안 발생하는 이벤트(LLM 호출, 도구 사용, 에이전트 전환 등)에 대한 콜백을 정의합니다.
AgentHooksBase — 에이전트(Agent) 수준의 Hook 기반 클래스. 개별 에이전트에 특화된 콜백
특정 에이전트 인스턴스에 직접 연결되는 콜백. agent.hooks 속성으로 설정합니다.
Python
from agents import Agent, Runner, RunHooksBase, AgentHooksBase
from agents.lifecycle import RunContextWrapper
import time
import json

# ─── Run 수준 Hook: 전체 실행에 걸친 관측/추적 ───
class ObservabilityHooks(RunHooksBase):
    """
    Run-level hooks: 전체 에이전트 실행을 관통하는
    관측 가능성(Observability) 콜백
    """

    async def on_agent_start(self, context: RunContextWrapper, agent):
        # 에이전트가 활성화될 때 호출
        # 멀티 에이전트에서 handoff(전환) 시마다 호출됨
        print(f"[START] Agent '{agent.name}' activated")
        context.context["start_time"] = time.time()

    async def on_llm_start(self, context, agent, system_prompt, input_items):
        # LLM API 호출 직전
        # system_prompt: 시스템 프롬프트 전문
        # input_items: 모델에 전달되는 메시지/도구 결과
        print(f"[LLM] Calling model for '{agent.name}'")
        print(f"  Input items: {len(input_items)}")

    async def on_llm_end(self, context, agent, response):
        # LLM 응답 수신 직후
        # response: ModelResponse 객체 (토큰 사용량 등 포함)
        usage = response.usage
        print(f"[LLM] Response: {usage.input_tokens}in / {usage.output_tokens}out")

    async def on_tool_start(self, context, agent, tool):
        # 로컬 도구(function tool) 실행 직전
        print(f"[TOOL] Starting: {tool.name}")
        context.context["tool_start"] = time.time()

    async def on_tool_end(self, context, agent, tool, result):
        # 도구 실행 완료 직후
        duration = time.time() - context.context.get("tool_start", 0)
        print(f"[TOOL] Finished: {tool.name} ({duration:.2f}s)")

    async def on_handoff(self, context, from_agent, to_agent):
        # Handoff: 에이전트 간 제어 이전 시
        # 멀티 에이전트 시스템에서 "바톤 터치" 시점
        print(f"[HANDOFF] {from_agent.name} → {to_agent.name}")

    async def on_agent_end(self, context, agent, output):
        # 에이전트가 최종 출력을 생성했을 때
        elapsed = time.time() - context.context.get("start_time", 0)
        print(f"[END] Agent '{agent.name}' finished in {elapsed:.2f}s")


# ─── Agent 수준 Hook: 특정 에이전트에만 적용 ───
class SafetyHooks(AgentHooksBase):
    async def on_tool_start(self, context, agent, tool):
        # 이 에이전트의 도구 호출만 검증
        if tool.name == "execute_sql":
            query = tool.input.get("query", "")
            if "DROP" in query.upper() or "DELETE" in query.upper():
                raise ValueError(f"위험한 SQL 차단: {query[:50]}")

# ─── 사용 ───
agent = Agent(
    name="database_agent",
    instructions="You are a helpful database assistant.",
    hooks=SafetyHooks(),   # Agent 수준 Hook
)

result = await Runner.run(
    agent,
    hooks=ObservabilityHooks(),  # Run 수준 Hook
    input="Show me the user table",
)

4.3 LangChain Callbacks & LangGraph Interrupt

LangChain Callback 시스템

LangChain의 콜백은 event-driven(이벤트 기반 — 특정 사건 발생 시 등록된 함수가 자동 호출되는 방식) 시스템으로, 3가지 핵심 추상화로 구성됩니다:

  • Handler (핸들러) — 실제 콜백 로직을 구현하는 클래스
  • Manager (매니저) — 여러 핸들러를 조정하고 이벤트를 전파하는 관리자
  • Configuration (설정) — 실행 체인을 통해 콜백을 전파하는 설정 객체
Python
from langchain.callbacks.base import BaseCallbackHandler

class AgentMonitor(BaseCallbackHandler):
    # ─── LLM 관련 콜백 ───
    def on_llm_start(self, serialized, prompts, **kwargs):
        print(f"[LLM 시작] 프롬프트 {len(prompts)}개")

    def on_llm_end(self, response, **kwargs):
        print(f"[LLM 완료] 토큰: {response.llm_output}")

    def on_llm_error(self, error, **kwargs):
        print(f"[LLM 에러] {error}")

    # ─── 도구(Tool) 관련 콜백 ───
    def on_tool_start(self, serialized, input_str, **kwargs):
        print(f"[도구 시작] {serialized['name']}: {input_str}")

    def on_tool_end(self, output, **kwargs):
        print(f"[도구 완료] 결과: {output[:100]}")

    # ─── 체인(Chain) 관련 콜백 ───
    def on_chain_start(self, serialized, inputs, **kwargs):
        print(f"[체인 시작] {serialized.get('name')}")

    def on_chain_end(self, outputs, **kwargs):
        print(f"[체인 완료]")

# 사용: 콜백을 config로 전달
result = agent.invoke(
    {"input": "질문"},
    config={"callbacks": [AgentMonitor()]}
)

LangGraph interrupt() — Human-in-the-Loop

Human-in-the-Loop (HITL) — 인간 개입 루프. 자동화된 프로세스의 특정 지점에서 인간의 판단/승인을 요구하는 패턴
고위험 결정(결제 처리, 데이터 삭제 등)에서 에이전트가 자동으로 실행을 중단하고, 인간의 승인을 받은 후 재개하는 패턴입니다.
Python
from langgraph.types import interrupt, Command
from langgraph.graph import StateGraph
from langgraph.checkpoint.memory import MemorySaver

def execute_payment(state):
    amount = state["amount"]
    recipient = state["recipient"]

    # interrupt()는 GraphInterrupt 예외를 발생시켜
    # 실행을 깨끗하게 중단하고 체크포인트에 상태를 저장
    approval = interrupt({
        "question": f"{recipient}에게 {amount}원을 송금하시겠습니까?",
        "amount": amount,
        "recipient": recipient,
    })

    # 인간이 승인하면 이 지점부터 재개
    if approval == "approved":
        return {"status": "결제 완료", "tx_id": process_payment(amount, recipient)}
    else:
        return {"status": "결제 취소됨"}

# Checkpointer(체크포인터): 그래프 상태를 저장하여
# 중단/재개를 가능하게 하는 영속성 메커니즘
graph = StateGraph(PaymentState)
graph.add_node("payment", execute_payment)
app = graph.compile(checkpointer=MemorySaver())

# 실행 → interrupt에서 자동 중단
result = app.invoke({"amount": 50000, "recipient": "홍길동"},
                    config={"configurable": {"thread_id": "tx-001"}})

# 인간이 승인 후 재개
result = app.invoke(
    Command(resume="approved"),
    config={"configurable": {"thread_id": "tx-001"}}
)

5.3 Layer 1: Next.js Middleware (HTTP/인프라 수준 Hook)

Next.js Middleware — Next.js의 미들웨어. 모든 요청이 라우트에 도달하기 전에 실행되는 코드. Edge Runtime에서 동작
인증 검증, Rate Limiting, 요청 로깅 등을 라우트 핸들러 도달 전에 수행합니다. AI 에이전트에 대한 "첫 번째 방어선"입니다.
중요한 구분 Next.js 공식 문서 경고: "Middleware is a UX optimization, not a security guarantee." (미들웨어는 UX 최적화이지, 보안 보장이 아닙니다.) 반드시 Server Component/Server Action에서 재검증해야 합니다. 미들웨어는 빠른 차단을, Route Handler는 완전한 검증을 담당합니다. 
src/middleware.ts
import { NextResponse } from 'next/server'
import type { NextRequest } from 'next/server'
import { Ratelimit } from '@upstash/ratelimit'
import { Redis } from '@upstash/redis'

// ─── Rate Limiter 설정 ───
// Sliding Window: 시간 창이 매 요청마다 슬라이딩되는 방식
// Token Bucket이 아닌 Sliding Window를 사용하는 이유:
// AI 요청은 처리 시간이 길어 버스트 허용보다 균일한 분배가 적합
const ratelimit = new Ratelimit({
  redis: Redis.fromEnv(),
  limiter: Ratelimit.slidingWindow(
    20,         // 20 requests
    '1 m'       // per 1 minute
  ),
  analytics: true,
  prefix: 'ai-agent',
})

export async function middleware(request: NextRequest) {
  // AI API 경로에만 적용
  if (!request.nextUrl.pathname.startsWith('/api/chat')) {
    return NextResponse.next()
  }

  // ─── 1. 인증 확인 ───
  const authHeader = request.headers.get('authorization')
  if (!authHeader?.startsWith('Bearer ')) {
    return NextResponse.json(
      { error: 'Unauthorized' },
      { status: 401 }
    )
  }

  // ─── 2. Rate Limiting ───
  const userId = extractUserId(authHeader)  // JWT에서 사용자 ID 추출
  const { success, limit, remaining, reset } = await ratelimit.limit(userId)

  if (!success) {
    return NextResponse.json(
      { error: 'Too many requests' },
      {
        status: 429,
        headers: {
          'X-RateLimit-Limit': limit.toString(),
          'X-RateLimit-Remaining': remaining.toString(),
          'X-RateLimit-Reset': reset.toString(),
          'Retry-After': Math.ceil((reset - Date.now()) / 1000).toString(),
        },
      }
    )
  }

  // ─── 3. 요청 로깅 ───
  const response = NextResponse.next()
  response.headers.set('X-Request-Id', crypto.randomUUID())
  response.headers.set('X-RateLimit-Remaining', remaining.toString())

  return response
}

// ══════════════════════════════════════════════════
// 목적: 모델 출력에서 민감 정보를 필터링하고,
// 위험한 내용을 차단합니다.

const SENSITIVE_PATTERNS = [
  { pattern: /\b\d{3}-\d{2}-\d{4}\b/g, replacement: '[SSN REDACTED]' },
  { pattern: /\b[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Z|a-z]{2,}\b/g,
    replacement: '[EMAIL REDACTED]' },
  { pattern: /\b\d{4}[\s-]?\d{4}[\s-]?\d{4}[\s-]?\d{4}\b/g,
    replacement: '[CARD REDACTED]' },
  { pattern: /\b01[0-9]-\d{3,4}-\d{4}\b/g,
    replacement: '[PHONE REDACTED]' },
]

function sanitizeText(text: string): string {
  let result = text
  for (const { pattern, replacement } of SENSITIVE_PATTERNS) {
    result = result.replace(pattern, replacement)
  }
  return result
}

export const guardrailMiddleware: LanguageModelV3Middleware = {
  // wrapGenerate: generateText() 호출을 감싸는 함수
  // doGenerate: 실제 모델 호출 함수. 이를 호출하여 결과를 받고 후처리
  wrapGenerate: async ({ doGenerate, params }) => {
    const result = await doGenerate()

    // 모델 출력에서 민감 정보 제거
    const sanitized = result.text ? sanitizeText(result.text) : result.text

    return { ...result, text: sanitized }
  },

  // wrapStream: streamText() 호출을 감싸는 함수
  // 스트리밍 청크를 하나씩 처리하여 실시간 필터링 가능
  wrapStream: async ({ doStream, params }) => {
    const { stream, ...rest } = await doStream()

    // TransformStream: 스트림 데이터를 변환하는 Web Streams API
    const transformStream = new TransformStream<
      LanguageModelV3StreamPart,
      LanguageModelV3StreamPart
    >({
      transform(chunk, controller) {
        if (chunk.type === 'text-delta') {
          controller.enqueue({
            ...chunk,
            delta: sanitizeText(chunk.delta),
          })
        } else {
          controller.enqueue(chunk)
        }
      },
    })

    return { stream: stream.pipeThrough(transformStream), ...rest }
  },
}

5.5 Layer 3: Agent Loop Callbacks (streamText / generateText)

Agent Loop — 에이전트 루프. LLM이 도구를 호출하고, 결과를 받아 다시 추론하는 반복 과정
streamText/generateText에 maxSteps를 설정하면, LLM → 도구 호출 → 결과 피드백 → LLM 재호출의 루프가 자동으로 실행됩니다.

6.2 Tool Gateway 패턴 — 도구 호출 중앙 검증

Tool Gateway — 도구 게이트웨이. API Gateway와 같은 개념을 도구 호출에 적용. 모든 도구 호출이 통과해야 하는 중앙 검증 계층
도구 허용 목록(allowlist), Rate Limiting, 범위 제한(scope restriction), 승인 게이트, 구조화된 로깅을 한 곳에서 처리합니다.
src/lib/ai/tool-gateway.ts
import { tool } from 'ai'
import { z } from 'zod'

// ─── 도구 호출 정책 정의 ───
interface ToolPolicy {
  maxCallsPerSession: number    // 세션당 최대 호출 횟수
  requiresApproval: boolean     // 인간 승인 필요 여부
  allowedScopes: string[]       // 허용된 범위 (예: 특정 테이블만)
  riskLevel: 'low' | 'medium' | 'high'
}

const TOOL_POLICIES: Record<string, ToolPolicy> = {
  getWeather:      { maxCallsPerSession: 10, requiresApproval: false,
                     allowedScopes: ['*'], riskLevel: 'low' },
  searchDatabase:  { maxCallsPerSession: 20, requiresApproval: false,
                     allowedScopes: ['users', 'orders', 'products'], riskLevel: 'medium' },
  sendEmail:       { maxCallsPerSession: 3,  requiresApproval: true,
                     allowedScopes: ['internal'], riskLevel: 'high' },
  deleteRecord:    { maxCallsPerSession: 0,  requiresApproval: true,
                     allowedScopes: [], riskLevel: 'high' },
}

// ─── Gateway 클래스 ───
export class ToolGateway {
  private callCounts = new Map<string, number>()
  private sessionId: string
  private userId: string

  constructor(sessionId: string, userId: string) {
    this.sessionId = sessionId
    this.userId = userId
  }

  // 도구 호출 전 검증
  async validate(toolName: string, args: Record<string, unknown>):
    Promise<{ allowed: boolean; reason?: string }> {

    const policy = TOOL_POLICIES[toolName]
    if (!policy) {
      return { allowed: false, reason: `알 수 없는 도구: ${toolName}` }
    }

    // 1) 호출 횟수 제한
    const count = this.callCounts.get(toolName) ?? 0
    if (count >= policy.maxCallsPerSession) {
      return {
        allowed: false,
        reason: `${toolName} 세션당 최대 ${policy.maxCallsPerSession}회 호출 가능 (현재: ${count})`,
      }
    }

    // 2) 범위 제한 (예: 특정 테이블만 허용)
    if (args.table && !policy.allowedScopes.includes('*')) {
      if (!policy.allowedScopes.includes(args.table as string)) {
        return {
          allowed: false,
          reason: `${toolName}은 ${policy.allowedScopes.join(', ')} 범위만 허용`,
        }
      }
    }

    // 3) 승인 필요 도구
    if (policy.requiresApproval) {
      const approved = await this.checkApproval(toolName, args)
      if (!approved) {
        return { allowed: false, reason: '인간 승인 대기 중' }
      }
    }

    // 호출 카운트 증가
    this.callCounts.set(toolName, count + 1)

    // 구조화된 감사 로그 (Audit Log)
    console.log(JSON.stringify({
      event: 'tool_gateway_pass',
      sessionId: this.sessionId,
      userId: this.userId,
      tool: toolName,
      args,
      callNumber: count + 1,
      riskLevel: policy.riskLevel,
    }))

    return { allowed: true }
  }

  private async checkApproval(toolName: string, args: Record<string, unknown>):
    Promise<boolean> {
    // 실제 구현에서는 WebSocket/Polling으로 관리자 승인 대기
    // 또는 큐에 넣고 비동기 처리
    return false
  }
}

// ─── Gateway를 적용한 도구 래퍼 ───
export function createGatedTool<T extends z.ZodObject<any>>(
  name: string,
  config: {
    description: string
    parameters: T
    execute: (args: z.infer<T>) => Promise<unknown>
  },
  gateway: ToolGateway,
) {
  return tool({
    description: config.description,
    parameters: config.parameters,
    execute: async (args) => {
      // 게이트웨이 검증을 도구 실행 전에 삽입
      const validation = await gateway.validate(name, args)

        throw new Error(`[GATEWAY DENIED] ${validation.reason}`)
      }
      return config.execute(args)
    },
  })
}

6.3 Proposal-Before-Execution 패턴

Proposal Before Execution — 실행 전 제안. 에이전트가 바로 행동하지 않고, 먼저 "이렇게 할 예정입니다"라고 구조화된 제안서를 제출하는 패턴
"Earn write access(쓰기 권한을 획득)" 모델 — 에이전트가 먼저 계획을 세우고, 검증을 통과한 후에만 실제 시스템에 대한 실행 권한을 얻습니다.

// ─── Step 2: 제안 검증 ───
async function validateProposal(proposal: {
  actionType: string
  target: string
  reason: string
  details: Record<string, unknown>
}): Promise<{ approved: boolean; reason: string }> {

  // 규칙 기반 검증 (Computational Guide)
  if (proposal.actionType === 'delete') {
    return { approved: false, reason: '삭제 작업은 자동 승인 불가' }
  }

  if (proposal.actionType === 'email' && !proposal.target.endsWith('@company.com')) {
    return { approved: false, reason: '외부 이메일 발송은 관리자 승인 필요' }
  }

  // AI 기반 검증 (Inferential Guide)
  const { text } = await generateText({
    model: anthropic('claude-haiku-4-5'),
    prompt: `다음 제안이 안전하고 적절한지 판단하세요.
제안: ${JSON.stringify(proposal)}
"APPROVED" 또는 "DENIED: 사유"로만 답변하세요.`,
  })

  if (text.startsWith('APPROVED')) {
    return { approved: true, reason: 'AI 검증 통과' }
  }

  return { approved: false, reason: text }
}

// ─── Step 3: 검증된 제안만 실행 ───
export async function executeWithProposal(userMessage: string) {
  // Phase 1: 에이전트가 제안 생성 (실행 없음)
  const proposalResult = await generateText({
    model: anthropic('claude-sonnet-4-5'),
    tools: proposalTools,
    maxSteps: 3,
    prompt: `사용자 요청: "${userMessage}"
이 요청을 처리하기 위해 필요한 행동을 proposeAction 도구로 제안하세요.
직접 실행하지 마세요.`,
  })

  // Phase 2: 제안 검증
  for (const step of proposalResult.steps) {
    for (const toolCall of step.toolCalls) {
      const validation = await validateProposal(toolCall.args)
      if (!validation.approved) {
        return { status: 'denied', reason: validation.reason }
      }
    }
  }

  // Phase 3: 승인된 제안만 실제 실행
  return executeApprovedProposals(proposalResult.steps)
}

6.4 prepareStep으로 동적 에이전트 루프 제어

prepareStep — 스텝 준비. 각 스텝 실행 전에 호출되어 해당 스텝의 설정을 동적으로 변경할 수 있는 Hook
이전 스텝의 결과에 따라 다음 스텝에서 사용할 도구, 모델 설정, 시스템 프롬프트 등을 동적으로 조정합니다.
src/lib/ai/dynamic-agent.ts
import { streamText, tool, stepCountIs } from 'ai'

const result = await streamText({
  model: anthropic('claude-sonnet-4-5'),
  messages,
  tools: { search, analyze, summarize, sendReport },

  // stopWhen: 에이전트 루프 종료 조건
  // stepCountIs(N): N번째 스텝 후 자동 종료 → 무한 루프 방지
  stopWhen: stepCountIs(10),

  // prepareStep: 각 스텝 실행 전에 호출
  // 이전 스텝 결과를 기반으로 다음 스텝 설정을 동적 변경
  prepareStep: async ({ stepNumber, previousSteps }) => {
    const lastStep = previousSteps[previousSteps.length - 1]

    // 패턴: 처음 3스텝은 검색 도구만, 이후 분석/요약 도구 추가
    if (stepNumber <= 3) {
      return {
        toolChoice: { type: 'auto' },
        activeTools: ['search'],     // 검색 단계
      }
    }

    // 검색이 충분하면 분석 단계로 전환
    if (stepNumber <= 6) {
      return {
        activeTools: ['search', 'analyze'],  // 분석 추가
      }
    }

    // 마지막 단계: 요약과 보고서 전송
    return {
      activeTools: ['summarize', 'sendReport'],
      toolChoice: { type: 'auto' },
    }
  },

  onStepFinish: ({ stepType, toolCalls }) => {
    console.log(`Step completed: ${stepType}, tools: ${
      toolCalls?.map(tc => tc.toolName).join(', ') || 'none'
    }`)
  },
})

6.5 클라이언트 사이드 Hook — useChat

useChat — Vercel AI SDK의 React Hook. 채팅 인터페이스의 전체 상태(메시지, 입력, 로딩, 에러)를 관리
서버와의 스트리밍 통신, 메시지 배열 관리, 폼 제출 처리를 자동화합니다. 클라이언트 사이드 도구 실행(onToolCall)도 지원합니다.
src/components/chat.tsx
'use client'

import { useChat } from '@ai-sdk/react'
import { useState } from 'react'

export function ChatInterface() {
  const [pendingApproval, setPendingApproval] = useState<null | {
    toolName: string
    args: Record<string, unknown>
    resolve: (approved: boolean) => void
  }>(null)

  const {
    messages,
    input,
    handleInputChange,
    handleSubmit,
    isLoading,
    error,
    addToolResult,
  } = useChat({
    api: '/api/chat',
    maxSteps: 5,

    // ─── 클라이언트 사이드 도구 실행 Hook ───
    // 서버가 아닌 클라이언트에서 실행해야 하는 도구
    // (예: 브라우저 위치 정보, 클립보드 접근)
    onToolCall: async ({ toolCall }) => {
      if (toolCall.toolName === 'getUserLocation') {
        // 브라우저 Geolocation API 호출
        const pos = await new Promise<GeolocationPosition>((resolve, reject) =>
          navigator.geolocation.getCurrentPosition(resolve, reject)
        )
        return { lat: pos.coords.latitude, lng: pos.coords.longitude }
      }

      if (toolCall.toolName === 'confirmAction') {
        // Human-in-the-Loop: 사용자에게 승인 요청
        return new Promise((resolve) => {
          setPendingApproval({
            toolName: toolCall.args.actionName,
            args: toolCall.args,
            resolve: (approved) => {
              setPendingApproval(null)
              resolve({ approved, confirmedAt: new Date().toISOString() })
            },
          })
        })
      }
    },

    // ─── 에러 핸들링 Hook ───
    onError: (error) => {
      console.error('Chat error:', error)
    },

    // ─── 응답 완료 Hook ───
    onFinish: (message) => {
      console.log('AI response complete:', message.id)
    },
  })

  return (
    <div>
      {/* 메시지 렌더링 */}
      {messages.map(m => (
        <div key={m.id}>{m.role}: {m.content}</div>
      ))}

      {/* Human-in-the-Loop 승인 UI */}
      {pendingApproval && (
        <div className="approval-dialog">
          <p>에이전트가 {pendingApproval.toolName} 실행을 요청합니다.</p>
          <pre>{JSON.stringify(pendingApproval.args, null, 2)}</pre>
          <button onClick={() => pendingApproval.resolve(true)}>승인</button>
          <button onClick={() => pendingApproval.resolve(false)}>거부</button>
        </div>
      )}

      {/* 입력 폼 */}
      <form onSubmit={handleSubmit}>
        <input value={input} onChange={handleInputChange} />
        <button type="submit" disabled={isLoading}>전송</button>
      </form>
    </div>
  )
}
Chapter 7

Anti-Pattern과 Best Practice

실패 사례에서 배우는 Hook 설계의 올바른 방향과 피해야 할 함정입니다.

7.1 주요 Anti-Pattern (피해야 할 패턴)

Anti-Pattern 1: Tool Bloat (도구 과잉) — 도구가 너무 많아 에이전트의 선택 품질이 저하되는 현상

에이전트에 30-50개 도구를 제공하면 성능이 급감합니다. 프로덕션에서는 20개 미만이 권장됩니다. Vercel은 80%의 도구를 제거하여 에이전트 성능을 향상시켰습니다.

해결: 도구를 카테고리별로 분리하고, prepareStep으로 단계에 따라 활성 도구를 동적으로 변경
Anti-Pattern 2: All-or-Nothing Autonomy (전부-아니면-전무 자율성) — 승인 게이트 없이 에이전트에 완전한 자율성을 부여하는 것

고위험 결정(결제, 삭제, 외부 커뮤니케이션)에는 반드시 Approval Gate(승인 게이트)를 배치해야 합니다. Replit의 DROP DATABASE 사건이 대표적 사례입니다.

해결: Tool Gateway의 requiresApproval 정책, LangGraph interrupt(), 클라이언트 onToolCall 승인 UI
Anti-Pattern 3: Compounding Error Cascade (복합 에러 연쇄) — 멀티 스텝에서 에러가 누적 곱으로 증폭되는 현상. 0.85^10 = 0.197

단계당 85% 정확도라도, 10단계 워크플로의 전체 성공률은 약 20%입니다.

해결: 각 스텝 후 검증 Hook(onStepFinish), 실패 시 조기 종료(stopWhen), 체크포인트 저장
Anti-Pattern 4: Hook Proliferation (Hook 비대화) — 너무 많은 Hook/규칙이 쌓여 오히려 성능을 저해하는 현상

HumanLayer는 AGENTS.md를 60줄 이내로 유지합니다. "Every line is competing for attention(모든 줄이 주의력을 두고 경쟁한다)".

해결: 각 Hook이 "특정 과거 실패 또는 외부 제약"에 근거하는지 정기적으로 검토. 모델이 개선되면 불필요한 Hook 제거
Anti-Pattern 5: Context Drift (컨텍스트 드리프트) — 기저 데이터가 변경되었는데 하네스가 이를 감지하지 못해 에이전트가 오래된 정보로 동작하는 현상

엔터프라이즈 AI 실패의 65%가 컨텍스트 드리프트에서 발생합니다. 스키마 변경, API 버전 업데이트, 비즈니스 규칙 변경 등이 원인입니다.

해결: FileChanged Hook으로 설정 파일 변경 감지, 정기적 하네스 감사(audit), 스키마 버전 관리

학술 연구 동향과 보안 고려사항

최신 학술 연구에서 드러난 AI 에이전트 Hook/보안의 미해결 과제와 연구 방향입니다.

8.1 최신 연구 동향 (2025-2026)

OWASP Top 10 for Agentic Applications (2025.12)

OWASP(Open Worldwide Application Security Project)는 2025년 12월, 도구 접근 권한을 가진 AI 에이전트에 특화된 Top 10 위험 목록을 발표했습니다. 이는 기존 LLM Top 10과 별도로, 에이전트의 새로운 공격 표면을 정의합니다.

  • Prompt Injection(프롬프트 인젝션 — 악의적 입력으로 에이전트의 행동을 조작)
  • Excessive Agency(과도한 권한 — 에이전트에 필요 이상의 도구/권한 부여)
  • Insecure Output Handling(안전하지 않은 출력 처리 — 에이전트 출력을 검증 없이 실행)
  • Supply Chain Attacks(공급망 공격 — 오염된 도구/플러그인을 통한 공격)

ETDI — 암호학적 도구 메타데이터 서명

ETDI (Extended Tool Definition Interface) — 확장 도구 정의 인터페이스. 도구 호출의 무결성을 암호학적으로 보장하는 프로토콜
도구의 스키마, 권한, 설명이 암호학적으로 서명되어, 도구가 중간에 변조되지 않았음을 검증할 수 있습니다. MCP(Model Context Protocol) 생태계에서의 도구 공급망 보안을 강화합니다.

비구성성(Non-compositionality) 문제

개별적으로 안전한 에이전트들이 조합되면 안전하지 않을 수 있습니다. arXiv 논문(2505.02077)에 따르면, 멀티 에이전트 시스템에서는:

  • Covert Collusion(은밀한 공모) — 에이전트들이 의도치 않게 협력하여 보안 정책을 우회
  • Cascading Failures(연쇄 실패) — 한 에이전트의 오류가 다른 에이전트로 전파
  • Emergent Behaviors(창발적 행동) — 개별 에이전트에서는 관찰되지 않던 예상치 못한 행동 출현
Hook 관점에서의 시사점 이는 개별 에이전트에만 Hook을 걸어서는 충분하지 않음을 의미합니다. 멀티 에이전트 시스템에서는 에이전트 간 상호작용에도 Hook이 필요합니다. Claude Code의 SubagentStart, SubagentStop, TaskCreated 이벤트가 이 문제를 해결하려는 초기 시도입니다.
