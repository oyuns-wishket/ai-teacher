Chapter 1

Session, Turn, Context Window

Agent가 동작하는 기본 실행 단위를 이해합니다.

AI 코딩 에이전트를 사용할 때 가장 먼저 이해해야 할 것은 실행 단위입니다. 모든 에이전트는 Session, Turn, Context Window라는 세 가지 계층 위에서 동작합니다.

Session (claude 실행 ~ 종료) ┌──────────────────────────────────────────────────┐ │ │ │ Turn 1: [CLAUDE.md + rules 로드] → 질문 → 응답 │ │ Turn 2: [CLAUDE.md + rules 로드] → 질문 → 응답 │ │ Turn 3: [CLAUDE.md + rules 로드] → 질문 → 응답 │ │ ... │ │ (컨텍스트 가득 차면 → 자동 Compaction) │ │ │ └──────────────────────────────────────────────────┘
개념설명비유
Session claude 명령어 실행부터 종료까지의 전체 단위. 대화 컨텍스트가 유지되는 범위. 브라우저 탭 하나를 여는 것
Turn 사용자 메시지 1개 + Claude 응답 1개 = 1턴. CLAUDE.md와 rules가 매 턴 주입됨. 채팅에서 한 번 주고받기
Context Window Claude가 한 번에 "볼 수 있는" 텍스트 총량 (약 200K 토큰). 시스템 프롬프트, 대화 히스토리, 파일 내용 등이 모두 들어감. 책상 위 작업 공간의 크기
Compaction 컨텍스트가 가득 차면 이전 대화를 자동 요약해서 압축. 세부 정보가 유실될 수 있음. 책상이 꽉 차면 노트를 요약본으로 교체
핵심 포인트 CLAUDE.md와 rules 파일은 "세션에 1회"가 아니라 "매 턴마다" 컨텍스트에 주입됩니다. 파일이 길수록 매 턴마다 토큰을 더 소비하며, 이는 곧 비용과 성능에 직결됩니다.
Compaction 주의 자동 요약은 편리하지만 세부 정보가 유실될 수 있습니다. 긴 작업은 중간중간 결과를 파일로 저장하는 습관이 중요합니다.
Chapter 2

설정 파일 계층 구조

에이전트에게 지시를 전달하는 파일들의 우선순위와 역할을 파악합니다.

우선순위

  1. 사용자 직접 지시 — 대화 중 입력 (최우선)
  2. CLAUDE.md / rules — 매 턴 로드 (파일 기반 지시)
  3. 시스템 기본 동작 — Claude의 기본 행동 패턴

파일 위치

~/.claude/ CLAUDE.md ← 글로벌 (모든 프로젝트 공통) settings.json ← 글로벌 설정 (권한, 환경변수) projects/<path>/memory/ ← 프로젝트별 메모리 (세션 간 유지) <git root>/ CLAUDE.md ← 프로젝트별 지시 (팀 공유) .claude/ CLAUDE.md ← 프로젝트별 지시 (대안 경로) rules/*.md ← 규칙 파일들 (전부 매 턴 로드) settings.json ← 프로젝트 설정 (git 공유용) settings.local.json ← 로컬 설정 (gitignore됨)
GLOBAL

~/.claude/CLAUDE.md

모든 프로젝트에 적용되는 개인 설정. 코딩 스타일, 선호 언어, 공통 컨벤션 등을 기록.

PROJECT

<git root>/CLAUDE.md

프로젝트 전용 설정. 빌드 명령어, 아키텍처 요약, 팀 컨벤션. git에 커밋하여 팀과 공유.

RULES

.claude/rules/*.md

카테고리별 규칙 파일. 모든 파일이 매 턴 전부 로드되므로 간결하게 유지.

LOCAL

.claude/settings.local.json

개인용 로컬 설정. API 키, 개인 환경변수 등. .gitignore에 추가하여 공유 방지.

상위 디렉토리는 거슬러 올라가며 로드 서브디렉토리에서 Claude를 실행하면, 작업 위치에서 상위 트리를 거슬러 올라가며 만나는 CLAUDE.md를 전부 읽어 이어붙입니다(concatenate, 덮어쓰기 아님). 루트에 가까운 파일이 먼저, 작업 위치에 가까운 파일이 나중에 로드됩니다. 작업 위치보다 아래쪽 서브디렉토리의 CLAUDE.md는 그 폴더의 파일을 읽을 때 지연 로드(just-in-time)됩니다. (자세히는 02. CLAUDE.md 세팅 가이드)
Chapter 3

CLAUDE.md & AGENTS.md 작성법

에이전트에게 효과적으로 지시하는 파일 작성 원칙

Anthropic과 OpenAI 모두 "사람이 직접 작성한 지시 파일이 LLM이 생성한 것보다 약 4%p 더 높은 성능을 보인다"는 실험 결과를 공유하고 있습니다. 자동 생성에 의존하지 말고, 직접 핵심만 기록하세요.

"각 줄마다 스스로 물어보세요: '이걸 제거하면 Claude가 실수할까?' 아니라면, 지우세요. 비대한 CLAUDE.md는 오히려 실제 지시를 무시하게 만듭니다." — Anthropic, Claude Code Best Practices
써야 하는 것
  • 빌드/테스트 명령어 (npm run dev, pytest)
  • 프로젝트 컨벤션 (네이밍, 코드 스타일)
  • 아키텍처 요약 (주요 디렉토리 역할)
  • 자주 쓰는 외부 서비스/API 정보
  • 팀 워크플로우와 PR 규칙
쓰면 안 되는 것
  • 코드에서 파악 가능한 것 (파일 구조, 의존성)
  • 지나치게 긴 설명 (매 턴 토큰 소비)
  • 일시적 작업 내용 (이건 memory 사용)
  • LLM이 자동 생성한 템플릿 그대로
  • git history에서 알 수 있는 정보
경험 법칙 CLAUDE.md는 500줄 이하, MEMORY.md는 200줄 이하로 유지하세요. /init 명령어로 초기 파일을 생성한 후, 점진적으로 다듬어가는 것이 좋습니다.

AGENTS.md (OpenAI Codex)도 동일한 원칙이 적용됩니다. 다만 Codex는 태스크형(fire-and-forget) 구조이므로, 대화보다는 명확한 지시 스타일로 작성하는 것이 효과적입니다.

Chapter 4

Prompt Cache & 토큰 경제학

비용을 좌우하는 캐싱 메커니즘과 토큰 최적화 전략

AI 에이전트의 비용 대부분은 입력 토큰에서 발생합니다. CLAUDE.md, rules, 대화 히스토리가 매 턴 전송되기 때문입니다. Anthropic의 Prompt Cache는 이 비용을 최대 90%까지 절감할 수 있습니다.

Turn 1 Turn 2 Turn 3 (5분 침묵) Turn N ┌──────┐ ───→ ┌──────┐ ───→ ┌──────┐ ───→ ┌──────┐ ───→ ┌──────┐ │전체 │ │캐시 │ │캐시 │ │캐시 │ │전체 │ │과금 │ │적중 │ │적중 │ │만료 │ │과금 │ │ │ │90%할인│ │90%할인│ │ │ │ │ └──────┘ └──────┘ └──────┘ └──────┘ └──────┘
항목수치의미
캐시 TTL5분마지막 사용 후 5분 내 재사용 시 캐시 적중
캐시 적중 할인90%입력 토큰 비용이 1/10로 감소
코드 토큰 밀도2.5~3 token/word일반 텍스트보다 2~3배 더 많은 토큰 소비
JSON 토큰 밀도3~5 token/word구조화 데이터는 토큰 소비가 가장 큼
멀티에이전트 오버헤드4~15x최적화 없이 멀티에이전트 사용 시 비용 폭증
평균 프롬프트 길이 (2024→2026)~1,500 → ~6,000 토큰에이전틱 워크플로우로 인해 4x 증가
"Lost in the Middle" 문제 2026년 4월 테스트 결과, 모든 LLM 모델에서 컨텍스트 중간에 위치한 정보의 정확도가 10~25% 하락하는 현상이 확인되었습니다. 중요한 지시는 CLAUDE.md의 상단 또는 하단에 배치하세요.
STRATEGY 1

CLAUDE.md 간결하게 유지

매 턴 전송되므로 짧을수록 비용 절감. 500줄 이하 권장.

STRATEGY 2

대화 간격 5분 이내

캐시 TTL 내에 다음 질문을 하면 90% 할인 유지.

STRATEGY 3

불필요한 파일 읽기 최소화

큰 파일을 통째로 읽으면 컨텍스트를 소비. 필요한 부분만.

STRATEGY 4

정적 프리픽스 보존

프롬프트 앞부분은 변경하지 마세요. 캐시 적중률의 핵심.

Chapter 5

Agent 아키텍처 패턴

LLM 에이전트의 주요 동작 패턴과 설계 원칙 — ReAct, CoT, Plan-Execute

현대 AI 에이전트는 단순한 Q&A를 넘어 추론(Reasoning)과 행동(Acting)을 반복하는 구조로 동작합니다. 2023년 Yao et al.의 ReAct 논문 이후, 이 패턴은 사실상 업계 표준이 되었습니다.

ReAct Pattern — Reasoning + Acting. Yao et al., Princeton/Google, 2023
LLM이 추론과 행동을 교차 수행하는 프레임워크. Thought → Action → Observation 루프를 반복하며 환각(hallucination)과 오류 전파를 줄입니다.
┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐ │ Thought │ ──→ │ Action │ ──→ │Observation│ ──→ │ Thought │ ──→ ... │ (추론) │ │(도구 실행)│ │(결과 관찰)│ │ (재추론) │ └──────────┘ └──────────┘ └──────────┘ └──────────┘

Claude Code가 파일을 읽고, 코드를 수정하고, 테스트를 돌리는 과정이 바로 이 Thought → Action → Observation 루프입니다.

패턴동작 방식적합한 상황
ReAct추론과 행동을 교대로 반복. "생각 → 실행 → 관찰" 루프.일반적인 코딩 작업, 디버깅, 탐색적 작업
Chain-of-Thought단계별로 추론을 전개한 후 최종 답변 도출.복잡한 논리 분석, 설계 결정, 코드 리뷰
Plan-then-Execute먼저 전체 계획을 수립하고, 별도의 실행기가 단계별 수행.대규모 리팩토링, 멀티파일 변경
Prompt Chaining하나의 출력을 다음 프롬프트의 입력으로 연결.데이터 파이프라인, 단계적 변환
Orchestrator-Worker오케스트레이터가 작업 분배, 워커들이 병렬 수행.독립적인 멀티파일 작업
Augmented LLM — Agent의 기본 구성 요소 에이전트 시스템의 기본 빌딩 블록은 Retrieval(정보 검색) + LLM(추론 엔진) + Tools(도구 실행)로 구성된 "증강된 LLM"입니다. 여기에 Memory, Context, Instructions가 추가되어 에이전트로 완성됩니다.
Chapter 6

Hooks & 자동화

특정 시점에 자동으로 스크립트를 실행하는 Hook 시스템

Hooks는 CLAUDE.md와 달리 "권고"가 아닌 "결정론적(deterministic)" 실행입니다. 특정 이벤트가 발생하면 지정된 쉘 명령어가 반드시 실행됩니다.

JSON
{
  "hooks": {
    "SessionStart": [
      { "command": "echo 'Session started at $(date)'" }
    ],
    "UserPromptSubmit": [
      { "command": "echo 'Turn received'" }
    ],
    "PostToolCall": [
      {
        "matcher": "Write",
        "command": "npx prettier --write $CLAUDE_FILE_PATH"
      }
    ]
  }
}
Hook실행 시점실행 횟수용도 예시
SessionStart세션 시작 시세션당 1회환경 체크, 컨텍스트 주입
UserPromptSubmit사용자 입력 시매 턴키워드 감지, 로깅
PreToolCall도구 실행 전매 호출위험 명령 차단
PostToolCall도구 실행 후매 호출자동 린트, 포맷팅
CLAUDE.md vs Hooks "코드 저장 후 항상 prettier를 돌려줘"라고 CLAUDE.md에 쓰면 가끔 무시될 수 있지만, Hook으로 설정하면 100% 실행이 보장됩니다. 반드시 실행되어야 하는 것은 Hook으로 설정하세요.
Chapter 7

Permission & Memory 시스템

권한 관리로 프롬프트 줄이기, 메모리로 세션 간 기억 유지하기

Permission 모델

자주 쓰는 안전한 명령어를 allowlist에 등록하면 매번 허용 프롬프트가 뜨지 않습니다.

JSON
{
  "permissions": {
    "allow": [
      "Read",
      "Bash(npm run *)",
      "Bash(git status)",
      "Bash(git diff *)",
      "Bash(npx prettier *)"
    ]
  }
}
꿀팁 /fewer-permission-prompts 명령어를 실행하면 사용 패턴을 분석하여 최적의 allowlist를 자동 추천해줍니다.

Memory 시스템

Memory는 세션을 넘어서 유지되는 기억입니다. Claude가 스스로 읽고 쓰는 파일 기반 시스템입니다.

~/.claude/projects/<path>/memory/ MEMORY.md ← 인덱스 (매 턴 로드, 200줄 이내) user_role.md ← 개별 메모리 (필요 시 로드) feedback_style.md ← 피드백 메모리 project_context.md ← 프로젝트 메모리
Memory 유형용도예시
user사용자 역할, 선호도, 지식 수준"시니어 백엔드 개발자, Go 전문"
feedback사용자가 준 피드백과 교정"테스트에서 DB 목킹하지 말 것"
project프로젝트 진행 상황, 목표"3/5 이후 머지 프리즈"
reference외부 시스템 위치 정보"버그 트래킹은 Linear INGEST"

Chapter 9

5가지 Workflow 패턴

PATTERN 1

Prompt Chaining

작업을 여러 단계로 나누고, 이전 출력을 다음 입력으로 연결. 각 단계에서 품질 게이트 가능.

PATTERN 2

Routing

입력을 분류하고 적절한 전문 처리기로 라우팅. 관심사의 분리.

PATTERN 3

Parallelization

독립적인 서브태스크를 동시에 실행하고 결과를 합산. 속도와 효율성.

PATTERN 4

Orchestrator-Workers

중앙 오케스트레이터가 작업 분배, 워커들이 수행. 프로덕션의 약 70% 차지.

PATTERN 5

Evaluator-Optimizer

생성기가 결과를 만들고 평가기가 피드백. 반복적 개선 루프.

Context Engineering & 도구 설계

Prompt Engineering을 넘어서는 새로운 패러다임

Anthropic은 2025년 공식 블로그에서 "Context Engineering"이라는 개념을 제시했습니다. 단순히 프롬프트를 잘 쓰는 것을 넘어, 에이전트에게 전달되는 전체 컨텍스트를 엔지니어링하는 것이 핵심입니다.

"원하는 결과의 가능성을 최대화하는 최소한의 고신호(high-signal) 토큰 집합을 찾아라." — Anthropic, Effective Context Engineering for AI Agents
PRINCIPLE 1

Context Rot 현상

토큰 수가 증가하면 성능이 오히려 저하됩니다. 더 많이 넣는다고 더 좋은 게 아닙니다.

PRINCIPLE 2

Just-In-Time 로딩

경량 식별자만 유지하고 런타임에 도구로 동적 로드. 미리 모든 것을 넣지 마세요.

PRINCIPLE 3

Lost in the Middle

LLM은 프롬프트의 시작과 끝 정보를 중간보다 잘 기억합니다.

PRINCIPLE 4

Sub-Agent 분리

전문 에이전트가 깨끗한 컨텍스트로 집중 작업 수행 후, 압축 요약만 반환.

도구(Tool) 설계 원칙

"에이전트-컴퓨터 인터페이스에 인간-컴퓨터 인터페이스만큼의 노력을 투자하라." — Anthropic, Effective Context Engineering
  • 도구 기능 간 겹침 최소화 — 모호성이 감소하면 정확도 향상
  • 각 도구는 자체 완결적이고 오류에 강건하게 — 명확한 단일 목적
  • 서술적이고 모호하지 않은 입력 파라미터 정의
  • 도구 집합 비대화 방지 — 너무 많은 도구는 결정 모호성을 유발
  • 모델의 인지 부하를 줄이는 형식 설계

Chapter 12

에이전트의 해부학 — 98%는 인프라다

2026년 학술 논문이 밝힌 Claude Code의 실제 구조

Dive into Claude Code — Liu et al., arXiv 2604.14228, 2026.04
Claude Code의 소스 코드를 분석한 학술 논문. 에이전트 코드의 98.4%는 인프라이며, 핵심 루프는 단순한 while문이라는 사실을 밝혔습니다.
Pseudo
// 에이전트의 핵심 = 단순한 while 루프 (전체의 ~1.6%)
while (task_not_complete) {
  response = llm.generate(context)      // 모델 호출
  if (response.has_tool_call) {
    result = execute_tool(response)    // 도구 실행
    context.append(result)              // 결과 추가
  }
}

// 나머지 98.4% = 이 루프를 둘러싼 인프라
// ├─ 권한 시스템 (7가지 모드 + ML 분류기)
// ├─ 컴팩션 파이프라인 (5계층 자동 압축)
// ├─ 메모리 관리 (3계층 아키텍처)
// ├─ 안전 장치 (위험 명령 감지/차단)
// ├─ 도구 오케스트레이션 (병렬 실행, 재시도)
// ├─ 캐싱 레이어 (프롬프트 캐시, 파일 캐시)
// └─ 사용자 인터페이스 (터미널 렌더링, 승인 UX)
논문의 핵심 결론 "기반 모델의 추론 능력이 수렴하면, 차별화 요소는 모델이 아니라 모델 주변의 결정론적 엔지니어링 하네스(harness)가 된다." 5가지 인간 가치가 13가지 설계 원칙을 이끈다고 분석했습니다.

개발자가 반드시 알아야 할 핵심 5가지

KEY 1

단순하게 시작하라

프레임워크 전에 직접 API 호출부터. 복잡성은 측정된 필요에 의해서만 추가.

KEY 2

컨텍스트는 유한한 자원

Context Rot, Lost in the Middle을 이해하고 JIT 로딩 + 컴팩션 전략 필수.

KEY 3

98%는 인프라다

에이전트 루프는 단순하지만, 주변 시스템(권한/안전/메모리/캐싱)이 진짜 차별화 요소.

KEY 4

캐싱은 게임 체인저

정적 프리픽스 보존으로 90%까지 비용 절감 가능.

KEY 5

도구 설계 = 에이전트 품질

겹치지 않고, 자체 완결적이며, 명확한 도구가 성능을 결정.

실전 체크리스트 & 흔한 실수

프로젝트 시작 전 확인할 것들과 많은 사람들이 빠지는 함정

세팅 체크리스트

  • CLAUDE.md 작성 (500줄 이내, 빌드/테스트 명령어 + 컨벤션)
  • .claude/rules/ 에 핵심 규칙 분리 (카테고리별 파일)
  • settings.json에 자주 쓰는 명령어 allowlist 추가
  • settings.local.json에 개인 환경변수 설정 (.gitignore됨)
  • 팀 공유용 settings.json은 git에 커밋
  • hooks로 린트/포맷 자동화 (PostToolCall 활용)
  • memory는 자연스럽게 축적 (강제로 채우지 않기)
  • /init 으로 초기 CLAUDE.md 생성 후 점진적 개선

흔한 오해와 현실

✗ "CLAUDE.md는 세션 시작 시 한 번만 읽힌다"
✓ 매 턴마다 읽힘 → 길수록 매 턴 비용 증가
✗ "rules 파일은 필요할 때만 선택적으로 로드"
✓ rules/ 디렉토리의 모든 파일이 매 턴 전부 로드
✗ "상위 폴더 CLAUDE.md는 무시되고 git root만 읽힌다"
✓ 작업 위치에서 상위 트리를 거슬러 올라가며 전부 읽어 이어붙임(덮어쓰기 X)
✗ "memory에 모든 것을 저장하면 좋다"
✓ MEMORY.md는 매 턴 로드 → 200줄 이내 유지 필수
✗ "세션 종료해도 대화가 유지된다"
✓ 세션 종료 = 컨텍스트 소멸. memory만 유지
✗ "Compaction은 손실 없이 안전하다"
✓ 요약이므로 세부 정보 유실 가능 → 중간 결과 파일 저장 권장
✗ "캐시는 항상 작동한다"
✓ 5분 TTL — 대화 간격이 길면 캐시 미스로 전액 과금
✗ "LLM이 생성한 CLAUDE.md가 더 완벽하다"
✓ 사람이 직접 작성한 것이 약 4%p 더 높은 성능을 보임
✗ "멀티에이전트가 항상 더 좋다"
✓ 최적화 없이 4~15x 비용 증가. 단일 에이전트로 시작 권장
✗ "에이전트 간 출력은 신뢰할 수 있다"
✓ 82.4%가 악성 도구 호출 그대로 실행 — 검증 필요
