Chapter 1

왜 지금 CLAUDE.md가 '대세'인가

설정 파일이 "있으면 좋은 것"에서 "팀의 인프라"가 된 배경

LLM은 세션 사이에 아무것도 기억하지 못합니다. 메모리 파일이 없으면 매번 같은 스택, 같은 컨벤션, 같은 프로젝트 맥락을 다시 설명해야 합니다. CLAUDE.md는 그 맥락을 모든 대화에 자동으로 진입시키는 유일한 메커니즘입니다.

"CLAUDE.md는 .gitignore에 던져두는 부산물이 아니라 인프라다. 버전 관리되고, PR에서 리뷰되고, 담당자가 있고, 에이전트 동작으로 테스트되어야 한다." — HumanLayer, Writing a good CLAUDE.md

표준화 타임라인 — 도구를 넘는 컨벤션

2025년 중반까지는 도구마다 형식이 파편화되어 있었습니다(.cursorrules, CLAUDE.md, .github/copilot-instructions.md …). 오픈소스 메인테이너는 거의 같은 파일을 5개씩 관리해야 했습니다. 그 후 빠르게 수렴했습니다.

Chapter 2

로딩 메커니즘 — 어떻게 읽히나

"어떻게 읽히는지"를 알아야 "어떻게 써야 하는지"가 보입니다. 가장 많이 오해하는 부분.

4계층 — 덮어쓰기가 아니라 '이어붙이기'

CLAUDE.md는 4개의 스코프(scope)에 존재할 수 있습니다. 핵심은 상위 파일을 하위 파일이 덮어쓰지 않는다는 점입니다. 발견된 모든 파일이 컨텍스트에 그대로 이어붙여(concatenate)집니다. 로드 순서는 넓은 스코프 → 좁은 스코프이며, 좁은(프로젝트) 지시가 넓은(사용자) 지시보다 뒤에 위치합니다.

스코프위치공유 범위
관리 정책
Managed policy		예) macOS /Library/Application Support/ClaudeCode/CLAUDE.md조직의 모든 사용자 (개인이 제외 불가)
사용자
User		~/.claude/CLAUDE.md나 자신 (모든 프로젝트)
프로젝트
Project		./CLAUDE.md 또는 ./.claude/CLAUDE.md팀 (git으로 공유)
로컬
Local		./CLAUDE.local.md나 자신 (현재 프로젝트만, .gitignore 권장)

상위 디렉토리 재귀 (Recursion)

Claude Code는 현재 작업 디렉토리에서 위쪽으로 트리를 거슬러 올라가며 CLAUDE.md를 모읍니다. 루트에 가까운 파일이 먼저, 작업 위치에 가까운 파일이 나중에 로드됩니다(= 더 "신선하게" 기억됨).

// foo/bar/ 에서 claude 실행 시 로드 순서 foo/CLAUDE.md ← 먼저 로드 (상위) foo/bar/CLAUDE.md ← 나중 로드 (작업 위치) foo/bar/CLAUDE.local.md ← 같은 폴더의 .local 은 맨 마지막
서브디렉토리 CLAUDE.md는 '지연 로딩' 작업 위치보다 아래쪽 서브디렉토리에 둔 CLAUDE.md는 실행 시점에는 로드되지 않습니다. Claude가 그 디렉토리의 파일을 실제로 읽을 때 함께 주입됩니다(just-in-time). 모노레포에서 모듈별 규칙을 둘 때 유용합니다.

언제 주입되나 — 그리고 어디에 주입되나

WHEN

매 세션 시작 시

CLAUDE.md와 auto memory는 모든 대화 시작 시점에 로드됩니다.

AFTER /compact

컴팩션 후 재주입

프로젝트 루트 CLAUDE.md는 /compact 후 디스크에서 다시 읽어 재주입됩니다(살아남음).

WHERE

system이 아니라 user 메시지

CLAUDE.md는 시스템 프롬프트의 일부가 아니라, 시스템 프롬프트 다음의 user 메시지로 전달됩니다.

HTML 주석은 토큰을 쓰지 않는다 (꿀팁) CLAUDE.md의 블록 단위 HTML 주석 <!-- 메인테이너 메모 -->는 Claude 컨텍스트에 주입되기 전에 제거됩니다. 사람용 메모를 토큰 소비 없이 남길 수 있습니다. 단, 코드 블록 안의 주석은 그대로 보존됩니다.
오해 주의 — "시스템 프롬프트 레벨"이 필요하면 CLAUDE.md는 user 메시지이므로, 진짜 시스템 프롬프트 레벨로 주입하려면 --append-system-prompt CLI 플래그를 써야 합니다. 매 실행마다 전달해야 하므로 대화형보다는 스크립트/자동화에 적합합니다.
Chapter 3

import(@)와 파일 분할

긴 지시를 쪼개는 법 — 그리고 "쪼개도 토큰은 안 줄어든다"는 함정

CLAUDE.md 안 어디서든 @경로 문법으로 다른 파일을 불러올 수 있습니다.

Markdown
# 프로젝트 개요는 README, 명령어는 package.json 참고
See @README for project overview and @package.json for npm commands.

# 추가 지시
- git 워크플로우 @docs/git-instructions.md
- 개인 메모 @~/.claude/my-project-instructions.md   # 홈 경로 OK
- 다른 도구와 공유 @AGENTS.md                       # Codex 등과 공용
규칙내용
최대 깊이import는 재귀적으로 다른 파일을 부를 수 있으나 최대 4 hops까지
상대 경로 기준작업 디렉토리가 아니라 import를 포함한 파일 기준으로 해석
경로 종류상대 경로 · 절대 경로 · 홈 경로(@~/...) 모두 허용
외부 import 승인프로젝트 밖 파일을 처음 import 하면 승인 대화상자가 뜸. 거절하면 비활성 유지
핵심 함정: import는 컨텍스트를 줄여주지 않는다 import된 파일은 그것을 참조한 CLAUDE.md와 함께 실행 시점에 전부 펼쳐져 로드됩니다. 즉 파일을 나누면 정리(organization)에는 도움이 되지만 토큰 절약은 0입니다. 진짜로 컨텍스트를 아끼려면 → 다음 장의 "온디맨드 로딩"을 보세요.

토큰을 진짜로 아끼는 분할: Skills / rules

@import — 항상 로드
  • 참조 즉시 launch 때 전부 펼쳐짐
  • 매 세션 토큰 비용 발생
  • "항상 필요한" 핵심 지시에만
Skill / path-scoped rule — 필요 시 로드
  • 호출되거나 매칭될 때만 로드(on-demand)
  • 평소엔 base 컨텍스트를 차지하지 않음
  • 특정 워크플로우(PR 리뷰, 마이그레이션)에 적합
설계 원칙 "PR 리뷰 절차", "DB 마이그레이션 순서"처럼 특정 상황에서만 쓰는 상세 지시는 CLAUDE.md(또는 @import)가 아니라 Skill이나 .claude/rules/의 path-scoped 규칙(YAML paths: 프론트매터)으로 빼세요. base 컨텍스트가 가벼워집니다.
Chapter 4

라인 수 최적화 — 왜 짧을수록 좋은가

"짧게 쓰라"는 취향이 아니라, 비용·성능·연구로 뒷받침되는 원칙입니다.

근거 1 — 토큰 경제학 (매 턴 비용)

CLAUDE.md는 정적 프리픽스의 일부로 프롬프트 캐시에 들어갑니다. 캐시가 따뜻할 때 읽기 비용은 기본 입력가의 10%(0.1x)로 저렴하지만, 이것이 "공짜"라는 뜻은 아닙니다.

항목수치의미
캐시 읽기(hit)기본가의 0.1x캐시가 살아있는 동안 매 턴 90% 할인
캐시 쓰기(5분)기본가의 1.25x첫 턴/캐시 만료 후엔 더 비쌈
캐시 TTL5분 (사용 시 갱신)5분 이상 침묵 → 캐시 증발, 다음 턴 전액 쓰기
손익분기읽기 1회5분 캐시는 단 1번 재사용해도 이득
최소 캐시 단위1,024 토큰 (Opus 4.8 / Sonnet 4.6)이보다 작으면 캐싱이 조용히 미적용
한글은 토큰을 더 먹는다 "1 토큰 ≈ 영어 4글자 ≈ 0.75단어"가 공식 추정치입니다. 그러나 한글(CJK)은 글자당 토큰이 훨씬 높아, 같은 의미라도 영어 대비 글자당 약 2~3배 토큰을 씁니다. 게다가 Opus 4.7+ 신형 토크나이저는 동일 텍스트에 최대 35% 더 많은 토큰을 쓸 수 있습니다. 한국어 CLAUDE.md는 라인 관리가 더 중요합니다.

근거 2 — Context Rot (길수록 멍청해진다)

"많이 넣을수록 좋다"는 직관은 틀렸습니다. 입력이 길어지면 단순 작업에서도 성능이 떨어진다는 것이 반복 검증되었습니다.

arXiv 2307.03172

Lost in the Middle

정보가 컨텍스트 중간에 있으면 정확도가 크게 떨어지고, 시작·끝에 있을 때 가장 높음(U자형). 긴 컨텍스트 모델도 동일.

Chroma 2025

Context Rot (18개 모델)

GPT·Claude·Gemini 등 18개 프런티어 모델 모두 입력이 길어질수록 성능 저하. Claude는 짧은(~300토큰) vs 긴(~113K) 프롬프트 격차가 컸음.

arXiv 2510.05381

길이 자체가 해롭다

관련 정보를 완벽히 검색해도, 입력 길이만으로 성능이 13.9~85% 저하. 산만요소가 아니라 "길이" 자체가 원인.

arXiv 2509.21051

지시가 많아질수록 (EMNLP'25)

동시 지시 개수가 늘수록 준수율이 일관되게 하락. 각각은 쉬운 지시도 한꺼번에 주면 못 지킴.

"트랜스포머는 n개 토큰에 대해 n² 쌍의 관계를 만든다. 컨텍스트가 커질수록 어텐션이 얇게 펴진다 … 원하는 결과의 가능성을 최대화하는 최소한의 고신호(high-signal) 토큰 집합을 찾아라." — Anthropic, Effective Context Engineering for AI Agents

형식과 강조

STRUCTURE

마크다운 구조

헤더·불릿으로 그룹화. Claude도 독자처럼 구조를 스캔한다. 빽빽한 문단보다 조직된 섹션.

EMPHASIS

IMPORTANT / YOU MUST

꼭 지켜야 할 규칙엔 강조어를 붙이면 준수율이 측정 가능하게 올라간다(공식 인정).

ORDER

명령어를 위로

빌드/테스트/실행 명령은 맨 위에. 모델은 프롬프트 경계(처음·끝)를 더 잘 본다.

VOICE

관찰이 아니라 지시문

"~인 것 같다"가 아니라 "X를 하라 / Y를 쓰지 마라" 같은 명령형으로.

Markdown · 좋은 예 (간결·구체)
# Code style
- 2-space 들여쓰기 (탭 금지)
- named export 사용, default export 금지

안티패턴 9가지

실무자들이 반복해서 경고하는 흔한 실수들

#안티패턴왜 나쁜가
1README/문서 통째 덤프가장 흔한 실패. 매뉴얼·튜토리얼·API 문서를 다 넣으면 절반은 무시됨
2뻔한 지시"클린 코드 작성", "시니어처럼 행동" — 토큰만 먹고 행동 변화 0
3오래된 내용(stale)코드는 바뀌는데 지시는 그대로 → 현실과 모순, 오히려 잘못된 판단 유발
4너무 길어 무시됨500줄+는 조용히 후순위로 밀림. "규칙 있는데도 안 지킨다" = 파일이 너무 길다는 신호
5모순되는 규칙상위/하위 파일이 충돌하면 Claude가 임의로 하나 선택(모노레포 주의)
6비밀키 포함git에 커밋되고 외부 도구가 읽음 — API 키·토큰·접속 문자열 절대 금지
7린터 대용으로 사용스타일 규칙은 지시로는 ~25~40%만 준수. ESLint/Prettier로 100% 강제하라
8인격 주입"신중하게 생각해" 류 — 실제 동작에 영향 없는 장식
9CLAUDE.local.md 커밋개인 샌드박스 URL·머신 설정이 팀에 새어나감 (.gitignore 필수)
특히 #6 — 비밀은 '위치'만 적어라 비밀 값 자체가 아니라, 비밀이 어디 있고 어떻게 접근하는지만 문서화하세요. 예: "DB 비밀번호는 1Password의 'staging-db' 항목 참조". CLAUDE.md는 평문으로 공유되는 파일입니다.
특히 #7 — 지시 vs 강제(Hook) AGENTS.md 가이드 연구에서 "지시만으로는 준수율 ~25~40%, PreToolUse Hook으로 강제하면 ~95%"로 보고됩니다. 반드시 지켜야 하는 것(포맷·금지 명령)은 CLAUDE.md가 아니라 Hook으로 — 이건 다음 자료(03. Hook 완전정복)의 주제입니다.

Chapter 8

검증과 반복 — 측정하고 개선하기

"CLAUDE.md는 코드처럼 다뤄라" — 리뷰하고, 가지치고, 동작으로 테스트한다

핵심 테스트: "이 줄을 지우면 실수하나?"

"각 줄마다 물어보라: '이걸 제거하면 Claude가 실수할까?' 아니라면 잘라라. 비대한 CLAUDE.md는 오히려 진짜 지시를 무시하게 만든다." — Anthropic, Claude Code Best Practices

두 가지 실패 신호 → 처방

증상 A — 규칙을 계속 어긴다

규칙이 있는데도 같은 잘못을 반복 → 파일이 너무 길어 그 규칙이 묻히는 중.

처방: 가지치기. 자명한 줄 삭제, 핵심 규칙을 위로, 강조어 추가.

증상 B — 답이 있는데 되묻는다

CLAUDE.md에 적힌 걸 Claude가 다시 질문 → 표현이 모호한 것.

처방: 구체화. "잘 해라"를 "X 명령을 실행한다"로.

언제 추가하나 — 일정이 아니라 '실패'에서

  • Claude가 같은 실수를 두 번째로 했을 때
  • 코드 리뷰가 Claude가 알았어야 할 것을 잡아냈을 때
  • 채팅에서 같은 교정을 두 번 입력했을 때
  • 새 팀원이 첫날 똑같은 맥락을 물어볼 만한 것일 때
사실보다 '교정'을 저장하라 실무자 보고에 따르면 단순 사실보다 "Y 말고 X를 해라, 왜냐하면 Z" 형태의 교정 메모가 훨씬 효과적입니다. 맥락(이유)이 붙은 교정은 복리처럼 쌓입니다.

도구로 검증하기

/memory

로드된 파일 감사

지금 세션에 실제 로드된 CLAUDE.md·local·rules를 전부 나열. 목록에 없으면 Claude는 그 파일을 못 본다.

/init

초안 생성·개선 제안

코드베이스를 분석해 시작 CLAUDE.md 생성. 이미 있으면 덮어쓰지 않고 개선안 제시.

관찰

동작이 실제로 바뀌나

수정 후 Claude의 행동이 바뀌는지 직접 확인. 변화가 없으면 그 줄은 효과가 없는 것.

auto memory

보완 루프

Claude가 스스로 학습(빌드 명령·선호)을 MEMORY.md에 기록(머신 로컬, 첫 200줄 로드). CLAUDE.md=사람 지시, auto memory=관찰 누적.

한 문장 요약 CLAUDE.md는 한 번 쓰고 끝나는 문서가 아니라 살아있는 코드입니다. git에 커밋해 팀과 함께 키우고, 문제가 생기면 리뷰하고, 정기적으로 가지치세요. 시간이 지날수록 가치가 복리로 쌓입니다.
