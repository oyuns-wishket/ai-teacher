Chapter 1

하네스 엔지니어링이란?

모델이 아니라 하네스가 진짜 경쟁력입니다. AI 에이전트를 제어하는 시스템 전체를 설계하는 방법론을 이해합니다.

"2025년은 AI 에이전트의 해였다면, 2026년은 에이전트 하네스(Harness)의 해입니다." 이제 누구나 LLM을 호출할 수 있고, 수십 개의 에이전트 프레임워크가 존재합니다. 그러나 실제 프로덕션에서 살아남는 팀은 더 좋은 모델을 쓰는 팀이 아니라 더 잘 설계된 하네스를 가진 팀입니다.

하네스(Harness)는 원래 말(馬)에 씌우는 마구(馬具)에서 온 단어입니다. 강력한 말의 에너지를 제어하고 원하는 방향으로 이끄는 도구 — AI 맥락에서도 정확히 같은 역할을 합니다. 모델이 아무리 강력해도 하네스 없이는 그 힘을 안전하게 활용할 수 없습니다.

에이전트 = 모델 + 하네스

Harness — 직역: 마구(馬具). AI 맥락에서는 모델을 감싸서 제어·관찰·안내하는 모든 인프라를 의미
프롬프트, 규칙 파일, Hook, CI/CD, 모니터링 등 모델 외의 모든 시스템. 에이전트가 무엇을 할 수 있고 없는지를 결정하는 제어 인프라 전체.
Harness Engineering — 하네스 엔지니어링. 에이전트의 제어 인프라를 체계적으로 설계·구축·운영하는 엔지니어링 분야
단순히 프롬프트를 쓰는 것이 아니라, 규칙·Hook·CI를 아우르는 전체 시스템을 엔지니어링하는 것. 하네스 엔지니어링의 목표는 모델의 능력을 안전하고 예측 가능하게 발휘시키는 것입니다.
"The agent is both the model and the harness. The harness is where you have the most leverage." — Birgitta Bockeler, Thoughtworks (Martin Fowler 블로그)
하네스는 단일 도구가 아닙니다 하네스는 하나의 도구가 아닙니다. 규칙 파일(CLAUDE.md, AGENTS.md), Hook(PreToolUse, PostToolUse), CI/CD 파이프라인, 모니터링, 권한 체계를 아우르는 전체 시스템입니다. 이 워크숍에서는 이 모든 것을 하나씩 직접 구축합니다.

확률적 준수 vs 결정론적 강제

Probabilistic Compliance — 확률적 준수. 프롬프트로 지시하면 모델이 대부분 따르지만, 100% 보장은 불가능한 방식
시스템 프롬프트에 "rm -rf를 실행하지 마세요"라고 쓰면 99% 따르지만, 1%의 실패가 치명적. 컨텍스트가 길어지면 지시사항을 "잊어버릴" 수 있고, Prompt Injection 공격에 취약합니다.
Deterministic Enforcement — 결정론적 강제. 시스템 수준에서 위반 자체를 물리적으로 불가능하게 만드는 방식
Hook으로 차단하면 모델이 무엇을 시도하든 100% 차단. 컨텍스트 길이나 프롬프트 인젝션과 무관하게 동작하는 시스템 수준의 강제입니다.
확률적 준수 (프롬프트만 사용)
시스템 프롬프트:
"절대로 rm -rf 명령을 실행하지 마세요.
 프로덕션 DB에 DROP TABLE을 실행하지 마세요."

→ 모델이 99% 따르지만, 1%의 실패가 치명적
→ 컨텍스트가 길어지면 "잊어버림"
→ Prompt Injection으로 우회 가능
결정론적 강제 (Hook 사용)
PreToolUse Hook:
if (command.includes("rm -rf")) {
  return { decision: "deny" };
}
if (command.includes("DROP TABLE")) {
  return { decision: "deny" };
}

→ 모델이 무엇을 시도하든 100% 차단
→ 컨텍스트 길이와 무관
→ Prompt Injection으로도 우회 불가능

3계층 아키텍처 개요

┌─────────────────────────────────────────────────────┐ │ ③ Quality Gate (차단) │ │ CI/CD · Stop Hook · 머지 방지 │ ├─────────────────────────────────────────────────────┤ │ ② Feedback Loop (교정) │ │ PostToolUse → 에러 재주입 → 자동 수정 │ ├─────────────────────────────────────────────────────┤ │ ① Constraint Harness (사전) │ │ 규칙 파일 · 린터 · PreToolUse │ ├─────────────────────────────────────────────────────┤ │ AI Model │ │ (여기는 여러분이 바꿀 수 없는 영역) │ └─────────────────────────────────────────────────────┘
Augment Code가 정의한 3계층 아키텍처 ① 사전에 규칙으로 제약(Constraint) → ② 실행 중 에러를 감지하고 교정(Feedback) → ③ 사후에 비준수 결과물을 차단(Quality Gate). 이 워크숍에서는 Chapter 2~4에서 각 계층을 차례로 구축합니다.

래칫 패턴

Ratchet Pattern — 래칫 패턴. 한 방향으로만 조이는 래칫 도구처럼, 실패가 발생하면 규칙과 Hook을 추가하여 같은 실수가 영구적으로 불가능해지는 개선 루프
에이전트가 실수를 저지를 때마다 하네스가 강화됩니다. 같은 실패는 두 번 다시 발생하지 않도록 시스템 수준에서 잠급니다.
실패 발생 ──→ 규칙 추가 (확률적) ──→ Hook 추가 (결정론적) ──→ 같은 실수 영구 차단 ↑ │ └──────────────── 새로운 유형의 실패 발생 ────────────────────────┘
Terminal Bench 실험 — Addy Osmani (Google Chrome 팀) 동일한 모델(Claude Sonnet)로 코딩 벤치마크를 실행했을 때, Hook과 규칙 파일만 교체하여 30위에서 5위로 점프했습니다. 모델을 바꾸지 않고도 하네스만으로 25단계 상승한 것입니다.

왜 하네스가 필수인가

실제 사고 사례 2025년 Replit Agent가 사용자 요청을 과잉 해석하여 프로덕션 데이터베이스에 DROP TABLE을 실행했습니다. PreToolUse Hook이 설정되어 있었다면 차단할 수 있었던 사고입니다.
통계 수치 출처
프로덕션 LLM 앱 프롬프트 인젝션 취약률 78% Robust Intelligence, 2025

Chapter 2

1계층: 규칙 파일 세팅

하네스의 첫 번째 계층입니다. CLAUDE.md와 AGENTS.md로 에이전트의 행동 규범을 정의합니다.

2.1 CLAUDE.md란?

CLAUDE.md — Claude Code가 세션 시작 시 자동으로 로딩하는 규칙 파일. 에이전트에게 "이 프로젝트에서는 이렇게 행동해라"를 선언하는 문서
빌드 명령, 코드 스타일, 금지사항 등을 한 곳에 정의. 에이전트의 "업무 매뉴얼"

CLAUDE.md는 계층 구조로 로딩됩니다. 아래 세 레벨이 모두 자동으로 적용되며, 하위 레벨이 상위 레벨을 오버라이드합니다.

  1. ~/.claude/CLAUDE.md — 글로벌 규칙 (모든 프로젝트 공통)
  2. 프로젝트루트/CLAUDE.md — 프로젝트 레벨 규칙
  3. .claude/rules/*.md — 세부 규칙 파일 (주제별 분리, 자동 로딩)
~/.claude/ ├── CLAUDE.md ← 글로벌 규칙 (모든 프로젝트에 적용) ├── settings.json ← Hook, 모델, 권한 설정 └── hooks/ ← Hook 스크립트 디렉터리 my-project/ ├── CLAUDE.md ← 프로젝트 규칙 (이 프로젝트만 적용) ├── .claude/ │ └── rules/ │ ├── testing.md ← 테스트 규칙 │ └── style.md ← 코드 스타일 규칙 └── AGENTS.md ← 범용 에이전트 규칙 (모든 AI 에이전트 지원)

2.2 좋은 CLAUDE.md 작성 원칙

  • 60줄 이내 권장 — 300줄 넘으면 토큰 낭비 + 중요 규칙 묻힘 (HumanLayer 가이드)
  • Progressive Disclosure 점진적 공개 — 핵심만 CLAUDE.md에, 세부사항은 rules/로 분리
  • 구체적 행동 지시 ("TypeScript strict mode 사용" ✅) vs 모호한 지시 ("좋은 코드 작성" ❌)
나쁜 CLAUDE.md
# My Project
코드를 잘 짜세요.
테스트를 작성하세요.
버그 없이 만드세요.
성능을 신경쓰세요.
보안에 주의하세요.
... (300줄 계속)

❌ 모호한 지시
❌ 300줄 = 토큰 낭비
❌ 중요한 규칙이 묻힘
좋은 CLAUDE.md
# My App

## 빌드
pnpm dev / pnpm build

## 스타일
- TypeScript strict, no `any`
- Server Components 기본

## 금지
- console.log 커밋 금지
- 환경변수 하드코딩 금지

✅ 40줄, 핵심만
✅ 구체적 행동 지시
✅ 세부사항은 rules/로
Anti-Pattern: CLAUDE.md 만능 파일 모든 규칙을 한 파일에 넣으면 토큰 낭비 + 중요한 규칙이 다른 내용에 묻힙니다. HumanLayer 연구에 따르면 60줄 이내가 최적입니다. 세부 규칙은 .claude/rules/로 분리하세요.

Chapter 3

2계층: Hook 시스템 구축

프롬프트는 '부탁'이고, Hook은 '법률'입니다. 결정론적 강제의 핵심인 Hook을 직접 만들어봅니다.

3.1 Hook 라이프사이클

Hook은 에이전트의 동작 흐름 중 특정 시점에 자동으로 실행되는 코드입니다. 아래 다이어그램은 세션 시작부터 종료까지 각 Hook이 언제 개입하는지를 보여줍니다.

세션 시작 │ SessionStart Hook ──→ 프로젝트 정보 주입 │ 사용자 메시지 ──→ 에이전트 판단 ──→ 도구 호출 결정 │ PreToolUse Hook │ │ allow deny ──→ 차단 (도구 실행 안 됨) │ 도구 실행 │ PostToolUse Hook ──→ 피드백 (에이전트에 재주입) │ 결과 반환 ──→ 다음 턴 또는 Stop │ Notification Hook

SessionStart는 세션이 열릴 때 한 번, PreToolUse는 도구 실행 직전, PostToolUse는 도구 실행 직후, Notification은 세션 종료 시 호출됩니다. 이 네 시점을 조합하면 에이전트의 전체 동작을 결정론적으로 제어할 수 있습니다.

3.2 5가지 핸들러 유형

Hook이 실행할 수 있는 핸들러— Handler. Hook이 트리거될 때 실제로 수행되는 동작의 유형는 5가지입니다.

유형 설명 사용 시점
command 셸 스크립트 실행 린트, 파괴적 명령 차단, 파일 검증
http HTTP 엔드포인트 호출 외부 서비스 연동, 웹훅 알림
mcp_tool MCP 서버의 도구 호출 MCP 기반 검증, 외부 시스템 연동
prompt LLM에 프롬프트 전달 AI 기반 판단 (코드 리뷰 등)
agent 서브에이전트 실행 복잡한 다단계 검증

3.3 settings.json 구조

Hook은 ~/.claude/settings.json 또는 프로젝트의 .claude/settings.json에 등록합니다. 기본 구조는 다음과 같습니다.

JSON{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "$HOME/.claude/hooks/block-dangerous.sh",
            "timeout": 5
          }
        ]
      }
    ]
  }
}
matcher — 매처. Hook이 실행될 도구를 지정하는 패턴. "Bash"는 Bash 도구에만, "Edit|Write"는 편집/쓰기 도구 둘 다에 적용. 정규식(RegExp) 지원

matcher는 도구 이름을 매칭하는 패턴입니다. timeout은 초 단위로, 시간을 초과하면 Hook이 무시됩니다. type은 위 5가지 핸들러 유형 중 하나를 지정합니다.

3.4 실습 3 — 파괴적 명령 차단 Hook

3
파괴적 명령 차단 Hook
목표: rm -rf /, DROP TABLE, git push --force main을 차단하는 PreToolUse Hook 셸 스크립트를 작성합니다.

Step A: 파일 생성

BASHmkdir -p ~/.claude/hooks
touch ~/.claude/hooks/block-dangerous.sh
chmod +x ~/.claude/hooks/block-dangerous.sh

Step B: 스크립트 작성

BASH#!/bin/bash
# PreToolUse Hook: 파괴적 명령 차단
# stdin으로 도구 호출 정보가 JSON으로 전달됨
INPUT=$(cat /dev/stdin)
COMMAND=$(echo "$INPUT" | jq -r '.tool_input.command // empty')

if [ -z "$COMMAND" ]; then
  exit 0
fi

BLOCKED=""

# rm -rf / 또는 rm -rf ~ (루트/홈 전체 삭제만 차단)
if echo "$COMMAND" | grep -qE 'rm\s+(-[a-zA-Z]*r[a-zA-Z]*\s+)+(-[a-zA-Z]*f[a-zA-Z]*\s+)*(\/$|\/\s|~\/$|~\/\s|~\s*$|\$HOME\/$|\$HOME\s)'; then
  BLOCKED="rm -rf on root/home directory"
fi

# rm -rf * (현재 디렉터리 전체 삭제)
if echo "$COMMAND" | grep -qE 'rm\s+(-[a-zA-Z]*r[a-zA-Z]*\s+)+(-[a-zA-Z]*f[a-zA-Z]*\s+)*\*'; then
  BLOCKED="rm -rf * (전체 삭제)"
fi

# DROP TABLE / DROP DATABASE
if echo "$COMMAND" | grep -qiE '\b(DROP\s+(TABLE|DATABASE|SCHEMA))\b'; then
  BLOCKED="DROP TABLE/DATABASE detected"
fi

# git push --force to main/master
if echo "$COMMAND" | grep -qE 'git\s+push\s+.*--force.*\s+(main|master)\b'; then
  BLOCKED="force push to main/master"
fi
if echo "$COMMAND" | grep -qE 'git\s+push\s+-f\s+.*\s+(main|master)\b'; then
  BLOCKED="force push to main/master"
fi

# git reset --hard
if echo "$COMMAND" | grep -qE 'git\s+reset\s+--hard\s+(origin|HEAD~)'; then
  BLOCKED="git reset --hard (destructive)"
fi

if [ -n "$BLOCKED" ]; then
  jq -n --arg reason "$BLOCKED" '{
    hookSpecificOutput: {
      hookEventName: "PreToolUse",
      permissionDecision: "deny",
      permissionDecisionReason: ("⛔ 차단됨: " + $reason)
    }
  }'
  exit 0
fi

exit 0

Step D: 검증

Claude Code에서 rm -rf /를 시도해보세요. Hook이 정상적으로 동작하면 "⛔ 차단됨: rm -rf on root/home directory" 메시지와 함께 명령이 차단됩니다.

디버깅 팁: claude --debug 플래그로 실행하면 Hook의 입출력을 실시간으로 확인할 수 있습니다. Hook이 0이 아닌 exit code를 반환하면 에이전트에 에러 피드백이 전달됩니다.
주의: Hook에서 exit 1을 반환하면 '에러'로 처리되어 에이전트가 재시도할 수 있습니다. '차단'하려면 반드시 exit 0 + JSON deny 응답을 사용하세요.

3.5 실습 4 — 자동 린트 피드백 Hook

4
자동 린트 피드백 Hook
목표: 파일 수정 후 자동으로 린트/타입체크를 실행하고, 에러를 에이전트에 재주입하여 자동 수정을 유도합니다.

Step A: 파일 생성

BASHtouch ~/.claude/hooks/post-edit-context.sh
chmod +x ~/.claude/hooks/post-edit-context.sh

Step B: 스크립트 작성

BASH#!/bin/bash
# PostToolUse Hook: 파일 수정 후 린트 피드백
INPUT=$(cat /dev/stdin)
FILE_PATH=$(echo "$INPUT" | jq -r '.tool_input.file_path // empty')

if [ -z "$FILE_PATH" ] || [ ! -f "$FILE_PATH" ]; then
  exit 0
fi

EXT="${FILE_PATH##*.}"
FEEDBACK=""

case "$EXT" in
  ts|tsx)
    PROJECT_DIR=$(dirname "$FILE_PATH")
    while [ "$PROJECT_DIR" != "/" ]; do
      if [ -f "$PROJECT_DIR/tsconfig.json" ]; then
        ERRORS=$(cd "$PROJECT_DIR" && npx --yes tsc --noEmit --pretty false 2>&1 | grep "error TS" | head -5)
        if [ -n "$ERRORS" ]; then
          FEEDBACK="TypeScript 에러 발견:\n$ERRORS"
        fi
        break
      fi
      PROJECT_DIR=$(dirname "$PROJECT_DIR")
    done
    ;;
  py)
    if command -v ruff &>/dev/null; then
      ERRORS=$(ruff check "$FILE_PATH" 2>&1 | head -5)
      if [ -n "$ERRORS" ]; then
        FEEDBACK="Ruff 린트 에러:\n$ERRORS"
      fi
    fi
    ;;
esac

if [ -n "$FEEDBACK" ]; then
  echo "$FEEDBACK"
  exit 0
fi

exit 0

PostToolUse 이벤트에 등록하며, matcher를 "Edit|Write"로 설정하여 파일 편집/쓰기 도구 모두에 적용합니다.

3계층: 품질 게이트

Hook이 실시간 방어라면, 품질 게이트는 최종 검문소입니다. CI/CD에서 에이전트의 결과물을 자동 검증합니다.

4.1 Quality Gate 개념

Quality Gate — 품질 게이트. 소프트웨어가 다음 단계(배포, 머지)로 넘어가기 전에 반드시 통과해야 하는 자동화된 검증 관문
CI에서 타입체크, 린트, 테스트가 모두 통과해야만 PR을 머지할 수 있는 시스템
Proposal-Before-Execution — 제안 후 실행 패턴. 에이전트가 먼저 계획을 제안하고, 시스템(또는 사람)이 검증한 뒤에야 실행을 허용하는 안전 패턴
"바로 실행"이 아니라 "제안 → 검증 → 실행" 3단계로 분리

Hook과 CI 품질 게이트는 서로 다른 시점에 작동하며, 각각의 역할이 있습니다. 아래 표는 세 가지 게이트를 비교합니다.

게이트 시점 차단 대상 예시
PreToolUse Hook 도구 실행 전 위험한 명령 rm -rf / 차단
Stop Hook 에이전트 루프 종료 시 미완성 결과 테스트 미통과 시 계속 작업 지시
CI Quality Gate PR/커밋 시 비준수 코드 타입 에러, 린트 실패, 테스트 실패
Defense in Depth— 심층 방어. 여러 계층의 보안을 중첩하여 단일 실패 지점을 제거하는 전략 — 세 게이트가 겹치는 것이 정상입니다. Hook이 놓친 것을 CI가 잡고, CI가 놓친 것을 코드 리뷰가 잡습니다.

4.2 Tool Gateway 패턴

Stripe MCP Toolshed 사례 — 400+ 도구를 관리하면서도 안전성을 유지하는 비결은 중앙 집중식 도구 게이트웨이입니다. 모든 도구 호출을 단일 진입점으로 라우팅하여 허용 목록 확인, 속도 제한, 위험도별 승인을 일관되게 적용합니다.

TYPESCRIPT// Tool Gateway: 모든 도구 호출이 이 게이트를 통과
function toolGateway(toolCall: ToolCall, context: AgentContext) {
  // 1. 허용 목록 확인
  if (!allowedTools.includes(toolCall.name)) {
    return { denied: true, reason: "허용되지 않은 도구" };
  }

  // 2. Rate Limiting
  if (rateLimiter.isExceeded(toolCall.name, context.sessionId)) {
    return { denied: true, reason: "호출 횟수 초과" };
  }

  // 3. 위험도별 승인
  const risk = classifyRisk(toolCall);
  if (risk === "dangerous") {
    return { requiresApproval: true, reason: "고위험 작업" };
  }

  return { allowed: true };
}
Tool Gateway — 도구 게이트웨이. 에이전트의 모든 도구 호출이 통과해야 하는 중앙 검문소. 허용 목록, 속도 제한, 승인 게이트를 한 곳에서 관리

4.3 실습 6 — GitHub Actions 품질 게이트

6
GitHub Actions 품질 게이트
목표: 에이전트가 만든 PR을 자동으로 검증하는 CI 파이프라인을 구성합니다.
YAML# .github/workflows/agent-qa.yml
name: Agent QA Gate

on:
  pull_request:
    branches: [main]

jobs:
  quality-gate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: '20'

      - name: Install dependencies
        run: npm ci

      - name: Type Check
        run: npx tsc --noEmit

      - name: Lint
        run: npx eslint . --max-warnings 0

      - name: Unit Tests
        run: npm test -- --coverage

      - name: Check for secrets
        run: |
          if grep -rn "PRIVATE_KEY\|SECRET_KEY\|password.*=" --include="*.ts" --include="*.js" src/; then
            echo "⛔ 하드코딩된 시크릿 발견!"
            exit 1
          fi

      - name: Bundle Size Check
        run: |
          npm run build
          SIZE=$(du -sk .next/static | cut -f1)
          if [ "$SIZE" -gt 500 ]; then
            echo "⚠️ 번들 크기 초과: ${SIZE}KB > 500KB"
            exit 1
          fi
핵심 포인트: GitHub Actions는 에이전트가 직접 수정할 수 없는 영역입니다. 리포지토리 설정에서 'Require status checks to pass'를 켜면, 에이전트가 CI를 우회하려 해도 물리적으로 머지가 불가능합니다. 이것이 결정론적 강제— Deterministic Enforcement의 진정한 힘입니다.
